Этот скилл собирает дневную порцию находок для одного радара проекта **Radar**.
Архитектуру и правила см. в корневом `CLAUDE.md`. Формат находки — `engine/schema/find.schema.json`.

## Вход

Аргумент — `slug` радара (напр. `claude-code`). Если не передан — спроси какой радар, либо
перечисли доступные: папки в `radars/`.

Все пути ниже относительно `radars/<slug>/`.

## Шаг 0. Загрузить контекст радара

Прочитай:
- `radars/<slug>/radar.config.json` → возьми `daily_target`, `freshness_days`, `taxonomy`, `platforms`, `language`.
- `radars/<slug>/sources.json` → структурированные источники (см. Шаг 1b). Если файла нет — работай только по WebSearch.
- `radars/<slug>/prompts/queries.md` → список поисковых запросов.
- `radars/<slug>/prompts/profile.md` → веса тегов, boost/mute.
- Существующие `radars/<slug>/data/finds/*.json` → собери множество уже известных `id` и `source_url` для дедупа.

Определи сегодняшнюю дату (формат `YYYY-MM-DD`) — это `date_found` и имя выходных файлов.

## Правило свежести (важно)

Берём только **свежие** новости: дата публикации оригинала (`published_at`) не старше
`freshness_days` дней от сегодня (по умолчанию 3). Окно = [сегодня − freshness_days; сегодня].
- Старше окна → отбрасываем, даже если находка хорошая.
- Если `published_at` определить не удалось — по умолчанию **не берём** (свежесть не подтверждена).
  Исключение: «вечнозелёные» ресурсы (curated GitHub-репо, офиц. докментация), которые
  обновляются постоянно — их можно взять, проставив `published_at` = дате последнего
  обновления/коммита, если она видна, иначе пометить `confidence: "low"` и не злоупотреблять.

## Шаг 1. Fan-out поиск (WebSearch)

Прогони `WebSearch` по запросам из `queries.md`. Приоритезируй запросы, относящиеся к темам
с высоким весом в `profile.md`; пропускай/понижай то, что в `mute`. Цель этого шага — собрать
пул кандидатов заметно больше, чем `daily_target` (напр. 40–60 ссылок), чтобы было из чего отбирать.
Где возможно, формулируй запросы на свежесть (последние дни/неделя), т.к. нужно окно `freshness_days`.

## Шаг 1b. Структурированные источники (sources.json)

Помимо WebSearch, пройди по `sources.json` — это прямые ленты (точные даты и ссылки).
Бери только `enabled: true`. Для каждого источника по `type`:

- **`youtube_rss`** — `WebFetch` URL ленты. Из Atom извлеки по каждому видео: `title`,
  ссылку (`watch?v=...`), `published` → `published_at`. Это РАБОЧИЙ канал (проверено).
  Отбирай видео, релевантные теме и попадающие в окно `freshness_days`. `source_platform: "youtube"`.
- **`rss`** — аналогично: `WebFetch`, парси entry (title/link/date). Если домен блокирует WebFetch — пропусти и отметь в резюме.
- **`reddit_json` / `reddit_rss`** — НЕ через WebFetch (Reddit его блокирует, и по IP тоже —
  см. `SOURCES.md`). Вместо этого запусти внешний сборщик:
  ```bash
  python3 engine/fetch_reddit.py <slug> --today <YYYY-MM-DD>
  ```
  Он сам читает `sources.json` (reddit_json, enabled), фильтрует по свежести и печатает
  кандидатов JSON в stdout — добавь их в общий пул. Если печатает `[]` и в stderr 403 —
  значит текущий IP в блоклисте Reddit; отметь это в резюме. С Reddit OAuth-ключами
  (env `REDDIT_CLIENT_ID`/`REDDIT_CLIENT_SECRET`) скрипт ходит через oauth.reddit.com и обходит блок.
  Посты не выдумывай — только то, что вернул скрипт.

Кандидатов из источников складывай в общий пул вместе с результатами WebSearch — дальше
единый дедуп и фильтрация. У источников обычно уже есть точная дата — это плюс к свежести.

## Шаг 2. Извлечение содержимого (WebFetch)

Для перспективных кандидатов вызови `WebFetch`, чтобы достать суть и подтвердить, что ссылка
рабочая и ведёт на оригинал. Извлеки: о чём пост/статья, кто автор, **дату публикации**
(`published_at`), что конкретно сделали с Claude Code (или с темой радара), какой результат.
Если страница недоступна/пустая — кандидат отбрасывается (правило: без рабочего `source_url` не публикуем).
**Если по странице видно, что новость старше окна `freshness_days` — отбрасываем сразу.**

> Не обязательно фетчить все 60 — фетчи самые многообещающие, пока не наберёшь надёжный пул ≥ daily_target.

## Шаг 3. Лёгкая фильтрация и отбор

- Выкинь: **несвежее** (старше окна `freshness_days` по `published_at`), рекламу/маркетинг без сути, дубли (по `id`/`source_url` против известных и внутри пула), нерелевантное теме.
- Приоритизируй по `profile.md`: находки с темами высокого веса — выше; `boost` — выше; `mute` — вон.
- Отбери ровно `daily_target` лучших (для claude-code = 15). Если качественных меньше — собери сколько есть и отметь это в дайджесте, не добивай мусором.

## Шаг 4. Сформировать находки по схеме

Для каждой отобранной находки построй объект строго по `engine/schema/find.schema.json`:
- `id` = sha1 от нормализованного `source_url` (нижний регистр, без utm/якорей). Бери первые 12–40 hex.
- `date_found` = сегодня.
- `title` — суть одной строкой, без кликбейта (8–140 симв.).
- `summary` — 2–3 предложения на языке радара (где/кто/как/результат), 40–600 симв.
- `details` — развёрнутый разбор для drill-down на сайте (markdown). При лёгкой фильтрации можно дать 3–6 предложений: что сделали, как воспроизвести, чем полезно.
- `tags` — 1–8 штук **только** из `taxonomy` радара.
- `source_url` — обязателен, рабочий.
- `source_platform` — из enum схемы.
- `published_at` — **дата публикации оригинала** (YYYY-MM-DD). Заполняй обязательно: на ней держится фильтр свежести и показ даты новости на сайте. Если точно неизвестна — см. «Правило свежести» (как правило, не берём).
- `author` — если известен.
- `confidence` — low/med/high по твоей оценке.

## Шаг 5. Записать файлы

1. `radars/<slug>/data/finds/YYYY-MM-DD.json` — JSON-массив находок. Если файл за сегодня уже есть — дополни/перезапиши, сохранив дедуп.
2. `radars/<slug>/data/digests/YYYY-MM-DD.md` — человекочитаемый дайджест.
   **Каждая находка идёт с двумя блоками: «Анонс» (суть, = summary) и «Расшифровка»
   (подробнее, о чём речь, = details). Не ограничиваться одним анонсом.**
   Формат (markdown, с пустой строкой после каждого заголовка — линтер MD022):
   ```
   # <title радара> — дайджест за YYYY-MM-DD

   _Собрано N находок. Под каждой: **Анонс** (суть) и **Расшифровка** (подробнее)._

   ---

   ## 1. <title>

   **Анонс.** <summary>

   **Расшифровка.** <details>

   `tag1 · tag2` · Источник: [<platform/author>](<source_url>)

   ---

   ## 2. ...
   ```
   Если у находки `details` пуст — на этапе записи дописать содержательную расшифровку
   (3–6 предложений: что сделали, как воспроизвести, чем полезно), а не оставлять только анонс.

## Шаг 6. Самопроверка (перед завершением)

- [ ] Кол-во находок = daily_target (или меньше с пометкой почему — напр. мало свежего за окно).
- [ ] У каждой непустой рабочий `source_url`.
- [ ] У каждой `published_at` в окне свежести [сегодня − freshness_days; сегодня].
- [ ] Все `tags` входят в `taxonomy`.
- [ ] Нет дублей по `id` внутри файла и против прошлых дней.
- [ ] JSON валиден относительно схемы.

В конце выведи краткое резюме: сколько собрано, разбивка по платформам/тегам, путь к файлам.

## Чего не делать

- Не выдумывать находки и не подставлять несуществующие ссылки.
- Не использовать платные API/ключи — только встроенные WebSearch/WebFetch.
- Не складывать данные в чужой радар.
