Этот скилл «обучает» тематику радара проекта **Radar**: превращает рейтинги пользователя в
обновлённый `profile.md`, который затем смещает будущий сбор (`collect-finds`).
Принцип — **прозрачность**: все изменения видны и редактируемы руками.

## Вход

Аргумент — `slug` радара. Пути относительно `radars/<slug>/`.

## Шаг 1. Загрузить данные

- `data/feedback/ratings.json` → карта `id -> rating` (1..5 звёзд, либо -1 = «мимо»/mute).
- Все `data/finds/*.json` → чтобы по `id` поднять находку (её `tags`, `author`, `source_platform`).
- `prompts/profile.md` → текущие веса и списки.
- `radar.config.json` → `taxonomy` (допустимые теги).

Если рейтингов нет — сообщи об этом и ничего не меняй.

## Шаг 2. Посчитать сигналы

Сопоставь каждый рейтинг с находкой и агрегируй:
- **По тегам:** средний рейтинг и число оценок на тег. Высокий средний (≥4) при ≥2 оценках → кандидат на повышение веса; низкий (≤2) → на понижение; -1 встречается часто → кандидат в mute.
- **По авторам / площадкам:** кто/что стабильно нравится (≥4) → кандидат в boost; стабильно «мимо» → в mute.

Будь консервативен при малом числе оценок (1–2): сдвигай веса мягко.

## Шаг 3. Обновить profile.md

- **Веса тегов:** сдвигай к сигналу плавно (напр. шаг ±0.1..±0.3 за прогон), держи в диапазоне 0.0–2.0. Не обнуляй тег с одной негативной оценкой.
- **Boost:** добавь авторов/темы/площадки со стабильно высоким рейтингом.
- **Mute:** добавь то, что стабильно «мимо». Mute сильнее веса — попавшее сюда сбор отсеивает.
- **Заметки об эволюции:** допиши строку: `- YYYY-MM-DD: <что изменил и почему (на основе N оценок)>`.

Сохраняй формат файла (таблица весов, секции Boost/Mute/Заметки) — не ломай структуру,
чтобы человек мог дочитать и поправить.

## Шаг 4. (Опционально) предложить правки queries.md

Если из сигналов явно следует новый удачный запрос или шумный запрос на удаление —
**предложи** правку `prompts/queries.md` пользователю, но не применяй без подтверждения.

## Шаг 5. Резюме

Выведи: сколько рейтингов обработано, какие веса изменились (было→стало), что добавлено в
boost/mute. Коротко объясни логику, чтобы пользователь мог скорректировать.

## Чего не делать

- Не менять веса резко по 1–2 оценкам.
- Не трогать `ratings.json` (он пишется сайтом, скилл его только читает).
- Не выходить за пределы `taxonomy` при работе с весами тегов.
