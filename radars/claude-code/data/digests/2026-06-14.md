## 1. Когда применять субагентов: правило «10+ файлов или 3+ независимых задачи»

**Анонс.** Builder.io и PubNub формулируют эвристику: направлять работу субагентам стоит, когда задача затрагивает 10+ файлов или содержит 3+ независимых куска работы. Субагенты выполняют побочную работу (исследование репо, прогон тестов, проверку) в своём контексте и возвращают только сводку, не засоряя основную сессию.

**Расшифровка.** Ключевой приём управления контекстом: вынести в субагента всё, что иначе залило бы основной диалог логами/выдачей поиска, которые больше не понадобятся. Для многошаговых пайплайнов субагентов выстраивают в цепочку с ролями (Product Spec → Architect → Implementer/Tester → QA) и связывают через хуки, получая воспроизводимый процесс. Субагенты подходят, когда исполнителям НЕ нужно общаться между собой; если нужна координация на лету — используются Agent Teams.

`subagents · workflows · automation` · Источник: [Builder.io](https://www.builder.io/blog/claude-code-subagents)

## 3. Хуки выросли в платформу автоматизации: 21 событие жизненного цикла

**Расшифровка.** Хуки — это shell-команды, срабатывающие в точках жизненного цикла, дающие детерминированный контроль (нужное действие происходит ВСЕГДА, а не по желанию модели). Настройка через /hooks или редактирование .claude/settings.json.

## 10. Plan Mode: «архитектурный режим» по Shift+Tab дважды для one-shot реализаций

**Анонс.** Приём: дважды нажать Shift+Tab, чтобы войти в Plan Mode — Claude продумывает запрос, не трогая файлы. Итерация над планом до удовлетворения часто позволяет модели реализовать задачу «с одного захода».

**Расшифровка.** Plan-режим разделяет мышление и исполнение: сначала довести план до ума без правок кода, потом дать реализовать. Снижает количество ошибочных правок и переделок. Один из самых высокорычажных приёмов для нетривиальных задач.

`tips · prompting · workflows` · Источник: [Charlie Hills](https://charliehills.substack.com/p/claude-code-beginner-advanced)

## 12. Skills vs Slash-команды: авто-вызов по контексту против ручного триггера

**Анонс.** Разбор различия: Skills авто-активируются, когда описание задачи совпадает с назначением скилла; слэш-команды требуют ручного запуска. Правило: Skills — для комплексных воркфлоу, Commands — для быстрых промптов и моментов, где важно, что человек сам выбирает время запуска.

**Расшифровка.** Важная заметка по безопасности продакшена: НЕ авто-триггерить разрушительные скиллы. Если воркфлоу может публиковать, деплоить, удалять, тратить деньги или трогать прод — оставляй явный шаг человеческого запуска (слэш-команда). Метод можно упаковать в скилл, но старт делать явным. Ровно этот принцип мы заложили в Radar: collect-finds/learn-profile запускаются вручную.

`skills · slash-commands · workflows` · Источник: [MindStudio](https://www.mindstudio.ai/blog/claude-skills-vs-slash-commands)

## 15. Hooks vs Slash Commands vs Skills: какой «поверхности» отдать воркфлоу

**Анонс.** Сравнение трёх механизмов расширения Claude Code (хуки, слэш-команды, скиллы) и того, какой из них должен владеть конкретным воркфлоу: детерминированные гарантии — хукам, ручной осознанный старт — командам, авто-подхватываемые комплексные процессы — скиллам.

**Расшифровка.** Помогает осознанно выбирать механизм под задачу, а не пихать всё в один. Хуки дают гарантию выполнения (не на усмотрение модели); команды — контроль момента запуска человеком; скиллы — авто-активацию по контексту. Полезно при проектировании собственной автоматизации (как в Radar).
