## 7. Справочник по хукам Claude Code 2026: PreToolUse, PostToolUse, Stop, SubagentStop с примерами

**Анонс.** Референс по хукам (2026): PreToolUse/PostToolUse/UserPromptSubmit/Stop/SubagentStop. Хук — shell-команда, читает JSON из stdin, сигналит кодом возврата и stdout-JSON. Практика: PostToolUse гоняет Prettier/тесты после каждой правки, exit 2 блокирует операцию и возвращает ошибку Claude. Детерминированность — модель не может «забыть» правило.

**Расшифровка.** Полный разбор lifecycle-хуков с примерами под Claude Code и Agent SDK. Суть: хук — это shell-команда (не промпт), читает JSON со stdin, отвечает через exit-код и stdout-JSON. Типовые сценарии: SessionStart подтягивает свежие баг-репорты; PostToolUse запускает форматтер (Prettier/Black) или тесты (pytest/npm test) после каждой правки файла; Stop пушит изменения в staging. Ключевое свойство — детерминизм: «PostToolUse → прогнать Prettier» срабатывает всегда, модель не может пропустить это, как бы ни рос контекст. Единственный способ обеспечить правило со 100% надёжностью в LLM-воркфлоу.

## 8. Skills vs MCP vs Plugins: когда что использовать (полный гайд 2026)

**Анонс.** Разбор различий: скилл — это инструкции (SKILL.md, ~30–50 токенов до вызова), MCP-сервер — это инструменты (доступ к внешним системам, но 50k+ токенов на 5 серверов), плагин — формат распространения (бандл скиллов + расширений). Рабочий паттерн 2026: 1 MCP на внешнюю систему + тонкие скиллы поверх.

**Расшифровка.** Практическая навигация по трём механизмам расширения. Скилл = инструкции (что делать), стоит ~30–50 токенов контекста, пока не вызван. MCP-сервер = инструменты (новые возможности: доступ к GitHub, Postgres, Linear, Sentry), но дорогой по контексту — 5 серверов могут съесть 50k+ токенов на старте. Плагин = единица распространения: бандлит скиллы, команды, агенты, хуки и MCP. Рекомендуемый сетап: 2–3 MCP-сервера (GitHub, Filesystem, один доменный) + несколько своих скиллов. Главный принцип: пин один MCP на внешнюю систему, а оркестрацию выноси в тонкие скиллы поверх.

`skills · mcp · tips · workflows` · Источник: [blog/Morph](https://www.morphllm.com/claude-code-skills-mcp-plugins)

## 9. Что изменилось в воркфлоу Claude Code 2026: естественный язык, CLAUDE.md, Plan Mode, длинные сессии

**Анонс.** Разбор актуальных практик: вместо громоздких промптов — простой естественный язык (инструмент сам читает файлы); обязательный CLAUDE.md в корне (entry points, деплой, ключевые файлы); Plan Mode перед мультифайловыми изменениями; длинные единые сессии вместо очистки контекста (авто-компакция копит знания).

**Расшифровка.** Четыре практических вывода: 1) Естественный язык вместо тяжёлого контекста — пиши «add source links to the email report», система сама читает файлы и сверяется с существующими паттернами. 2) CLAUDE.md в корне репо — entry points, процедуры деплоя, расположение ключевых файлов; стабильная инициализация сессии. 3) Plan Mode (/plan) перед изменениями, затрагивающими много файлов или важные решения — создаёт контракт до кода. 4) Длинные единые сессии вместо обнуления контекста между задачами — авто-компакция прозрачно держит лимиты, накопленные знания ускоряют последующие задачи.

## 12. Видео: тест вложенных субагентов в Claude Code — оркестратор, который не трогает код

**Анонс.** Owain Lewis (17 июня) на практике гоняет вложенные субагенты: наверху оркестратор, который управляет воркфлоу и сам никогда не редактирует код, а правки делегирует субагентам-исполнителям. Разбор паттерна manager/worker для безопасной автономной работы.

**Расшифровка.** Свежий практический разбор вложенных субагентов (до 5 уровней). Ключевой приём: верхний агент-оркестратор держит план и координацию, но не трогает файлы напрямую — иначе он смешивает планирование и исполнение и теряет контроль над контекстом. Реальные правки уходят субагентам-воркерам с ограниченными правами. Полезно тем, кто строит глубокие деревья делегирования: видно, как не дать дереву агентов «расползтись» и как разделить ответственность manager↔worker.
