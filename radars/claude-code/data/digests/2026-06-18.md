## 2. Microsoft: уязвимость Claude Code GitHub Action раскрывала ANTHROPIC_API_KEY — фикс в 2.1.128

**Анонс.** Microsoft Threat Intelligence (5 июня) разобрала уязвимость в Claude Code GitHub Action: через вредоносный промпт в issue/комментарии агент читал /proc/self/environ инструментом Read (он не был в Bubblewrap-изоляции, в отличие от Bash) и сливал ANTHROPIC_API_KEY. Anthropic закрыла дыру в v2.1.128, заблокировав доступ к чувствительным /proc-файлам.

**Расшифровка.** Показательный кейс безопасности агентов в CI/CD. Механика: атакующий внедряет промпт через недоверенный GitHub-контент (тело issue, PR, комментарии); агент, обрабатывая его, читает /proc/self/environ через инструмент Read — и тот, в отличие от Bash (Bubblewrap + scrubbing окружения), не был изолирован, поэтому ANTHROPIC_API_KEY утекал. Исследователи обошли и safety-фильтры модели, и сканер секретов GitHub, оформив кражу как «compliance review» и попросив усечь префикс ключа. Таймлайн: отчёт 29 апреля, митигация 5 мая (v2.1.128 — блок доступа к чувствительным /proc-файлам). Вывод для всех, кто гоняет Claude Code в Actions: обновляться и не давать агенту обрабатывать недоверенный ввод с доступом к секретам.

**Расшифровка.** Полезная сводка по тому, куда уехал MCP, и что из этого работает в Claude Code: транспорт Streamable HTTP стал стандартом для удалённых облачных серверов (вместо HTTP+SSE); аутентификация — OAuth 2.1 (с июньской спеки 2025), то есть нормальные токены вместо простых API-ключей; реестр официальных MCP-серверов уже >9400, MCP нативно поддерживают Salesforce Agentforce, Cloudflare, Replit, Azure OpenAI. Важная для практики деталь: Claude Code понимает уведомления list_changed — сервер может на лету добавлять/менять инструменты, промпты и ресурсы без disconnect/reconnect. Ориентир при выборе и деплое своих MCP-серверов.

## 7. Context engineering вместо prompt engineering: 6 опор и правила хорошего CLAUDE.md

**Анонс.** Гайд по «инженерии контекста»: CLAUDE.md грузится в контекст на каждом запросе, поэтому держи его коротким и плотным (<~200 строк) — команды build/test/lint, конвенции, архитектура в 3 предложения, жёсткие ограничения и подводные камни. Разница с prompt engineering: не «хорошо спросить», а «дать нужную информацию в нужный момент».

**Расшифровка.** Систематизация того, как кормить Claude Code контекстом. CLAUDE.md — файл, который читается в начале каждой сессии; каждая строка стоит токенов на каждом запросе, поэтому правило — короткий и сигнал-плотный файл (<~200 строк): точные команды сборки/тестов/линта, конвенции (нейминг, обработка ошибок), архитектура в три предложения, жёсткие ограничения и известные «грабли». Для Enterprise CLAUDE.md кэшируется (первый запрос платит полную цену, последующие в пределах ~5 минут — по сниженной). Практики: /init для стартового файла, /clear при переключении на несвязную задачу (CLAUDE.md несёт durable-контекст дальше), и трактовать файл как feedback loop — дописывать правила, когда видишь пробелы в выводе.

## 9. Полный гайд: хуки, субагенты и скиллы Claude Code — когда что применять

**Анонс.** Разбор трёх слоёв расширяемости Claude Code: хуки (детерминированное правило кодом), субагенты (изолированная параллельная работа в своём контексте) и скиллы (переиспользуемые инструкции, грузятся только при вызове). С чёткими правилами выбора: slash-команда / скилл / субагент / хук под конкретную задачу.

**Расшифровка.** Практическая навигация по расширяемости: хук — когда нужно правило, которое модель не может «забыть» (срабатывает кодом); субагент — для изолированной/параллельной работы, его подробный вывод не засоряет основной контекст, возвращается только итог; скилл — когда есть доменная логика или вспомогательные файлы, тело грузится лишь при вызове (дёшево по контексту). Правило выбора: slash-команда для шаблона промпта, скилл для доменной логики, субагент для изолированной параллельной работы, хук для детерминированного контроля. Старт рекомендуют со скиллов — проще всего и сразу полезно.

## 14. Практика: субагенты Claude Code в проде — как делегировать и не потерять контроль

**Анонс.** Прикладной гайд (Nimbalyst) по субагентам в реальной работе: субагент — это специалист со своим системным промптом, набором инструментов и правами; его многословный вывод (поиск файлов, дампы логов) остаётся изолированным, а в основную беседу возвращается только итог. Разбор, когда делегировать и как сохранять управляемость.

**Расшифровка.** Практический разбор «как это работает у нас в проде»: субагент изолирует контекст — его шумная работа (grep по файлам, длинные логи, многошаговое рассуждение) не засоряет главную сессию, наверх идёт только сводка. Это и экономит контекст, и повышает управляемость. Гайд даёт ориентиры: для какой работы заводить субагента (изолированная, параллельная), как давать ему узкие права инструментов, как не уйти в неуправляемое дерево делегирования. ## 15. Кейс TDD: как Claude Code пишет тесты первыми (Red-Green-Refactor) на реальном воркфлоу

**Анонс.** Разбор TDD-воркфлоу на Claude Code: попросить написать падающие тесты для ещё не существующей фичи (чтобы модель не наплодила mock-заглушек), довести реализацию до зелёного, отрефакторить. PostToolUse-хук гоняет линтер и тесты после каждой правки. Есть готовый skill test-driven-development, делающий дисциплину автоматической.

**Расшифровка.** Практика «тесты вперёд» с конкретикой: ключ — явно просить Claude написать тесты для функциональности, которой ещё нет (иначе он склонен застабить воображаемый код). Цикл: написать падающий тест (Red) → минимальная реализация до прохода (Green) → рефакторинг при зелёных тестах → 100% green перед PR. Автоматизация: PostToolUse-хук запускает линтер/тесты после каждой правки файла — мгновенная обратная связь без блокировки агента. Готовый skill test-driven-development (и tdd-writer) выдаёт полный тест-файл в фреймворке проекта (Jest/pytest/Go testing/RSpec), упорядочивая happy path → edge cases → error scenarios. Превращает TDD из намерения в автоматическую практику.
