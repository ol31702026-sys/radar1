## 1. Anthropic: 7 способов «рулить» Claude Code — CLAUDE.md, rules, skills, subagents, hooks, output styles, system prompt

**Анонс.** Официальный гайд Anthropic (18 июня) систематизирует семь механизмов управления поведением Claude Code и объясняет, когда какой брать: CLAUDE.md — базовые факты о репозитории, rules — путь-специфичные ограничения, skills — процедуры, subagents — изолированные задачи, hooks — детерминированная автоматизация, output styles — смена роли, system prompt — добавочные стандарты.

**Расшифровка.** Anthropic раскладывает «штурвал» Claude Code по слоям. **CLAUDE.md** — фундамент (build-команды, структура каталогов, конвенции); держать <200 строк и перезагружать при компакции. **Rules** — ограничения для конкретных путей или сквозных тем; грузятся только когда релевантны, экономя токены против always-on инструкций. **Skills** — процедурные воркфлоу (деплой, релиз-чеклист, ревью), которые Claude вызывает по требованию. **Subagents** — побочные задачи в отдельном окне контекста, возвращают только итог. **Hooks** — всё детерминированное (линтер после правок, постинг в Slack). **Output styles** — для серьёзной смены роли. **System prompt append** — добавить стандарты, не трогая ядро роли.

## 2. Исследование Anthropic: в Claude Code решает доменная экспертиза, а не стаж программиста

**Анонс.** Anthropic проанализировала ~400 000 сессий Claude Code от ~235 000 пользователей (окт 2025 – апр 2026). Доля «проверенно успешных» сессий растёт с 15% у новичков до 28–33% у экспертов; непрограммисты-доменные эксперты почти догоняют инженеров (26% против 30%). Решают две воспроизводимые привычки: точные инструкции и перенаправление агента вместо отказа при ошибке.

**Расшифровка.** Работа «Agentic coding and persistent returns to expertise» (16 июня). Ключевое: успех определяется не общим скиллом кодинга, а доменной экспертизой в конкретной задаче. Цифры: у новичков 19% сессий бросают после ошибки, у средних/экспертов — 5–7%. Одна экспертная сессия порождает ~12 действий и ~3200 слов вывода — впятеро больше, чем у новичка. Чем больше экспертизы человек проявляет, тем больше работы Claude делает на одну инструкцию.

## 3. Claude Code v2.1.178: команды-агенты стали неявными и появился синтаксис прав Tool(param:value)

**Анонс.** В релизе v2.1.178 (16 июня) убраны инструменты TeamCreate/TeamDelete — теперь на сессию подразумевается неявная команда агентов. Добавлен синтаксис правил доступа вида Tool(param:value) (например Agent(model:opus)), подхватываются скиллы из вложенных каталогов .claude/skills, а auto mode оценивает субагента перед запуском.

**Расшифровка.** Изменения упрощают мульти-агентную работу: вместо явного создания/удаления команды она теперь подразумевается на сессию. Новый синтаксис правил `Tool(param:value)` позволяет точечно разрешать/запрещать вызовы по параметрам (напр. ограничить субагентов конкретной моделью). Скиллы из вложенных `.claude/skills` теперь загружаются, `/doctor` показывает плоское дерево, плюс правки багов в субагентах. Как воспроизвести: обновить Claude Code до ≥2.1.178 и проверить новые правила прав в settings.

## 5. Как заставить Claude Code держать TDD: хуки + CLAUDE.md для цикла red-green-refactor

**Анонс.** Карлос Олива Паскуаль (11 июня) показывает, как сделать TDD обязательным, а не опциональным: правило в CLAUDE.md «сначала падающий тест, запусти npm test, убедись что КРАСНЫЙ, и только потом реализация» плюс PostToolUse-хук, прогоняющий тесты после каждой записи файла. Тезис: «промпт ограничивает намерения Claude, хуки ограничивают его действия».

**Расшифровка.** Проблема: без принуждения LLM пишет реализацию первой, а тесты подгоняет под неё — и они ловят меньше багов. Решение из трёх частей: (1) строгое правило процесса в CLAUDE.md; (2) хук PostToolUse в .claude/settings.json, который автоматически запускает `npm test` после каждой записи файла — Claude сразу видит красный/зелёный; (3) опора на ограничения процесса, а не напоминания. Воркфлоу: падающий тест → минимум кода до зелёного → рефактор с проверкой тестами → защита от регрессий.

`testing · hooks` · Источник: [blog/Carlos Oliva Pascual](https://dev.to/stacknotice/claude-code-tdd-force-red-green-refactor-with-hooks-claudemd-2026-26gk)

## 6. Субагенты Claude Code: изоляция контекста, экономия на моделях и параллельность

**Анонс.** Разбор (9 июня): каждый субагент получает только явно переданный контекст и свежее окно — меньше «замусоренного» контекста и выше точность. Лёгким задачам можно назначить дешёвую модель (Haiku), а дорогую оставить оркестратору; параллельный fan-out/fan-in срезает время выполнения на 60–80%.

**Расшифровка.** Практические принципы работы с субагентами. **Изоляция контекста:** субагент видит только то, что передал родитель, что убирает нерелевантный шум и повышает точность. **Стоимость:** не каждой задаче нужна самая дорогая модель — маршрутизация лёгких задач на Haiku заметно режет расход. **Параллелизм:** независимые субагенты бегут одновременно, ускоряя работу на 60–80% против последовательного выполнения, но нужно аккуратно разводить зависимости и следить за рейт-лимитами. Рекомендуемый паттерн — fan-out/fan-in.
