## 1. /design-sync: двусторонняя синхронизация Claude Design и Claude Code

**Анонс.** Июньский апдейт Claude Design добавил команду /design-sync — двусторонний мост с Claude Code: pull тянет дизайн-систему и экраны в репозиторий, push отправляет собранный в коде интерфейс обратно на канвас для визуальной доводки.

**Расшифровка.** Обновление Claude Design (18 июня 2026) превращает прототайпер в рабочий инструмент: импортируемые дизайн-системы (токены, гайдлайны, библиотеки компонентов из git или файлов), WYSIWYG-редактирование прямо на канвасе вместо чата, экспорт в PDF/PowerPoint/Canva и десктоп на всех платных планах. Ключевое для разработчика — команда `/design-sync`: pull подтягивает дизайн-систему и выбранные экраны в кодовую базу для реализации против реальных компонентов, push синхронизирует уже написанный код обратно в Claude Design. Это закрывает петлю «дизайн → код → дизайн» и держит бренд-консистентность между дизайнерами и инженерами.

## 2. cc-session-recover: восстановление сессий Claude Code после упора в квоту

**Анонс.** Опенсорс-утилита softcane автоматически возобновляет сессию Claude Code, прерванную лимитом/квотой: пишет HANDOFF.md, ждёт сброса квоты по медленному расписанию и сама подхватывает работу с места остановки. Квоту не обходит.

**Расшифровка.** cc-session-recover (github.com/softcane/cc-session-recover) решает боль долгих сессий, которые теряют контекст при упоре в rate-limit. Два режима: heartbeat держит активную сессию живой, пока терминал открыт; closed-terminal watcher возобновляет сессию по ID через headless-команды. Во время блокировки утилита пишет файл восстановления HANDOFF.md, ретраит по медленному расписанию и автоматически продолжает после сброса квоты. README прямо подчёркивает: инструмент не обходит лимиты, а лишь автоматизирует возобновление. Полезно тем, кто гоняет длинные автономные прогоны и не хочет вручную восстанавливать прогресс.

## 3. Как писать циклы в Claude Code: /loop, /goal и динамические воркфлоу

**Анонс.** Гайд показывает, что рутинные действия после ответа агента (тест → ревью → коммит → ждать фидбэк) — это уже циклы, и их стоит вернуть самому агенту. Разбор трёх механик: /loop по времени, /goal по условию и Claude-спроектированные динамические воркфлоу.

**Расшифровка.** Идея: возьми то, что человек делает руками после вывода Claude, — это и есть тело цикла. Любой цикл состоит из триггера (интервал или условие завершения), промпта с действиями и условия остановки. Три приёма: (1) `/loop 5m <промпт>` — запуск по расписанию, «Claude просыпается, что-то делает, отчитывается и идёт снова» (напр. мониторинг PR на новые комментарии ревью); (2) `/goal` — агент работает, пока отдельный оценщик не подтвердит измеримое условие, останавливается сам; (3) динамические воркфлоу — попросить Claude сам спроектировать структуру цикла (например «чини падающие тесты по одному на PR с проходами ревью») и показать план до выполнения. Практично для автономных прогонов без ручного протаскивания контекста.

## 4. «Код — новый сервер, спецификации — новый Terraform»

**Анонс.** Daniel Stolf доказывает: с удешевлением генерации кода в version control стоит коммитить не код, а спецификации — как Terraform сделал источником правды конфиги, а не серверы. Описывает фреймворк Speccraft со спеками как первоклассными артефактами.

**Расшифровка.** Тезис: коммитя код, мы коммитим вывод процесса, а критическое мышление теряется выше — в тикетах и переписке. Эволюция повторяет инфраструктуру: было «серверы → инфра-код», стало «код → спецификации»; код — это build-артефакт, регенерируемый из намерения, а спеки хранят решения, ограничения и обоснования. Speccraft (на базе Claude Code Agent Skills): спеки — версионируемые артефакты репозитория; код генерируется структурированным процессом из спек; тесты выводятся из спецификаций ДО реализации; хуки блокируют правки прод-кода без соответствующих изменений тестов; решения и конвенции привязаны к спекам и переживают сессии. Ревью смещается с деталей реализации на вопрос «имеет ли смысл новая спека и соответствует ли ей код».

`skills · prompting` · Источник: [blog/Daniel Stolf](https://dev.to/dcstolf/code-is-the-new-server-specs-are-the-new-terraform-l4h)

## 5. Урок: Claude Code с --dangerously-skip-permissions попал в прод-БД

**Анонс.** Автор в 3 часа ночи включил --dangerously-skip-permissions, и из-за неверной ссылки на env-файл Claude прогнал миграции D1 не на staging, а на проде. Данные уцелели (ADD COLUMN), но автор перестроил защиту: allowlist в settings.json, отдельный worktree и правила в CLAUDE.md.

**Расшифровка.** Разбор реального инцидента и того, как теперь устроена безопасность. Флаг `--dangerously-skip-permissions` убирает единственный момент, когда человек видит команду перед запуском. Три слоя защиты: (1) allowlist в `.claude/settings.json` — `"allow": ["Bash(wrangler d1 execute * --local*)"]`, `"deny": ["Bash(wrangler d1 execute *)"]`; (2) отдельный git worktree только со staging-кредами, физически изолирующий доступ к проду; (3) инструкции в `CLAUDE.md` требовать подтверждения перед касанием прода (автор отмечает: в длинных сессиях это слабеет по мере роста контекста). За две недели после внедрения Claude не запустил ни одной прод-команды; три события deny — все из-за неоднозначных промптов самого автора при переключении контекста, а не сбоев модели.

`automation · tips` · Источник: [blog/riversea](https://dev.to/riversea/i-let-claude-code-run-dangerously-skip-permissions-on-my-production-db-heres-what-i-changed-4p8)
