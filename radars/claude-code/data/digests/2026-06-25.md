Плюс мелочи UX: фильтрация статусов задач в workflows, клики мышью по меню в fullscreen, исправления --resume, structured output и таймаутов MCP-инструментов.

## 2. Claude как agent provider в JetBrains IDE (GitHub Copilot, public preview)

**Анонс.** GitHub Changelog (22 июня): Claude доступен в public preview как провайдер агента внутри GitHub Copilot для JetBrains IDE. Ставите Claude Code CLI, прописываете путь в настройках и выбираете Claude в пикере агентов в панели Copilot Chat. Пока работает в режиме bypass-permissions, настраиваемые разрешения обещают позже.

**Расшифровка.** Анонс расширяет присутствие Claude в IntelliJ/PyCharm/WebStorm и других JetBrains IDE: вместо отдельного плагина Claude встраивается как agent provider в Copilot Chat. Сценарий: установить Claude Code CLI, указать путь к нему в настройках Copilot, затем выбрать Claude в выпадающем списке агентов и вести сессию прямо из IDE. Важное ограничение превью — агент работает в bypass permissions mode (без гранулярного контроля над действиями), настраиваемые разрешения заявлены в будущем апдейте. Полезно тем, кто живёт в JetBrains и не хочет переключаться в терминал.

`ide-integration` · Источник: [blog/GitHub](https://github.blog/changelog/2026-06-22-new-features-and-claude-as-agent-provider-preview-in-jetbrains-ides/)

## 4. session-orchestrator: loop-engineering для кодинг-агентов (research → plan → wave-execute)

**Анонс.** GitHub (★36): community-плагин превращает ad-hoc сессии в повторяемый цикл research → plan → wave-execute → close с гейтами верификации. Работает поверх Claude Code, Codex CLI, Cursor и Pi. Лицензия MIT.

**Расшифровка.** Это попытка формализовать «loop engineering» — вместо одного длинного промпта агент гоняется по фиксированным фазам: исследование кодовой базы, планирование, волновое выполнение задач и закрытие с проверкой результата на каждом гейте. Плагин host-agnostic: один и тот же цикл запускается в Claude Code, Codex CLI, Cursor и Pi. Полезен тем, кто хочет повторяемости и контроля качества в длинных автономных прогонах, а не «как получится». Хорошо ложится на dynamic workflows и субагентов.

`workflows · subagents` · Источник: [github/Kanevry](https://github.com/Kanevry/session-orchestrator)

## 5. claude-handoff-revive: восстановление сессии Claude Code из handoff-файла

**Анонс.** Show HN: инструмент сохраняет состояние сессии Claude Code в handoff-файл — авто-сохранение, снапшоты и шеринг через PR, чтобы продолжить работу после вылета/компакта или передать контекст коллеге.

**Расшифровка.** Проблема: при компакте, исчерпании лимита или вылете теряется наработанный контекст сессии. Решение — handoff-файл, в который складывается состояние; его можно авто-сохранять, делать снапшоты и шарить через pull request. По сути это «git для состояния агента»: вернуться к прошлой точке или передать сессию другому человеку/машине. Удобно для длинных задач и командной работы, где одну ветку ведут по очереди.

`automation · tips` · Источник: [hn/sofumel](https://github.com/sofumel/claude-handoff-revive)

## 7. universal-mcp-sync (usync): локальный менеджер MCP-конфигов для нескольких AI-клиентов

**Анонс.** GitHub: usync — local-first менеджер конфигурации MCP для тех, кто использует несколько AI-клиентов (Claude Code, Cursor и др.). Держит единый список MCP-серверов и синхронизирует его между клиентами из одного источника правды.

**Расшифровка.** Боль: один и тот же набор MCP-серверов приходится прописывать руками в каждом клиенте, и конфиги разъезжаются. usync хранит конфигурацию локально в одном месте и раскатывает её в Claude Code, Cursor и другие клиенты, чтобы везде был одинаковый набор инструментов. Local-first — данные не уходят в облако. Полезно тем, кто работает в нескольких агентах и устал синхронизировать MCP вручную.

`mcp` · Источник: [github/nawodyaishan](https://github.com/nawodyaishan/universal-mcp-sync)

## 14. Vibe Under Constraint: Claude Code пишет доказательства на Lean 4

Вывод обобщается: чем сильнее внешний верификатор (типы, тесты, компилятор), тем полезнее автономный агент. Хороший пример того, как дисциплина инструментов важнее объёма промпта.
