## 2. InfoQ: как Claude сам строит исполнительные «харнессы» для динамических воркфлоу

**Анонс.** Разбор InfoQ (15 июня) о том, как Claude Code динамически генерирует JavaScript-харнессы, координирующие множество независимых агентов: fan-out-and-synthesize, adversarial verification, турнирные воркфлоу и роутинг моделей по стоимости задачи.

**Расшифровка.** Robert Krzaczyński объясняет внутреннюю механику: вместо одного контекстного окна Claude пишет оркестрационный скрипт, раздаёт подзадачи специализированным агентам и проверяет результат. Ключевые стратегии: «fan-out-and-synthesize» (параллельные подзадачи с последующей сборкой), «adversarial verification» (агенты-ревьюеры оспаривают выводы других), турнирные схемы (несколько агентов решают одну задачу разными способами и соревнуются) и роутинг моделей (дешёвые модели на простые стадии, мощные — на сложное рассуждение). Отдельно разобраны риски: «агентная лень», self-preferential bias и goal drift — и как мультиагентная координация их смягчает.

## 3. Developers Digest: практический гайд по динамическим воркфлоу Claude Code

**Анонс.** Гайд (11 июня) с конкретикой по dynamic workflows: как их запускать (ultracode / «use a workflow» / /effort ultracode), когда использовать pipeline() против parallel(), зачем нужна опция schema и как делать прогоны воспроизводимыми через args.

**Расшифровка.** Материал переводит динамические воркфлоу в рабочие приёмы. Запуск: вставить «ultracode» в промпт, попросить «use a workflow» обычным языком, либо задать /effort ultracode на уровень сессии. Главные примитивы: pipeline() — дефолт для потоковой обработки (элемент A может быть на стадии 3, пока B ещё на стадии 1), а parallel() — только когда стадии реально нужны все предыдущие результаты сразу. Опция schema форсирует structured-output вызов: валидация на уровне tool-call, модель сама ретраит при несоответствии. Для детерминизма таймстемпы передаются через args, а не Date.now(), потому что рантайм журналирует вызовы для resume.

`workflows · automation` · Источник: [blog/Developers Digest](https://www.developersdigest.tech/blog/claude-code-dynamic-workflows-guide)

## 5. Рефакторинг с Claude Code: воркфлоу «правило одного», который реально работает

**Анонс.** Статья (18 июня) описывает дисциплину рефакторинга из 4 шагов (Понять → Спланировать → Делать кусками → Проверить) и «правило одного»: один файл, одно изменение, один шаг, один тест за раз, с коммитом после каждого зелёного.

**Расшифровка.** Carlos Oliva Pascual формулирует методологию против «большого рефакторинга в один присест». Шаги: (1) Understand — расспросить кодовую базу о зависимостях и рисках до правок; (2) Plan — попросить Claude предложить стратегию без написания кода; (3) Execute in Chunks — по одному маленькому шагу с прогоном тестов после каждого; (4) Verify — проверить, что ничего не упущено. Базовый принцип — «the rule of one». Подчёркивается полный контекст через CLAUDE.md, чтение всех зависимых файлов перед изменением и коммит после каждого успешного шага.

`refactoring · workflows` · Источник: [blog/stacknotice](https://dev.to/stacknotice/refactoring-with-claude-code-the-workflow-that-actually-works-2026-3gg7)
