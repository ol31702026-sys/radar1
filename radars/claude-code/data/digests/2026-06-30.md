## 1. teng-lin/notebooklm-py — неофициальный Python-API для Google NotebookLM

**Зачем.** Если вы из Claude Code прогоняете исследования и хотите автоматически сгружать источники в NotebookLM, генерировать из них подкаст/конспект и забирать результат — без ручного клика по веб-интерфейсу. Сценарий: агент собрал десяток ссылок, сам создал notebook, задал вопросы, скачал аудио-саммари.

**Как.** Ставится как CLI: `uv tool install "notebooklm-py[browser]"` (или `pipx install "notebooklm-py[browser]"`); как библиотека — `uv add notebooklm-py`. Для Claude Code есть готовый скилл: `notebooklm skill install`, а инструкции для агента — `notebooklm agent show claude`.

**Подвох.** Это обёртка над недокументированным внутренним API Google: автор прямо пишет «APIs may break — Google can change internal endpoints anytime», проект не аффилирован с Google и рекомендован «для прототипов, исследований и личных проектов», не для прода. Требует логина в Google-аккаунт через браузер, при активном использовании Google может троттлить. Звёзд много (★16988), но стабильности это не гарантирует — завязка на чужой нестабильный backend.

## 2. Памяти модели не нужно — нужен контекст ситуации

**Зачем.** Полезно, если вы строите для Claude Code слой памяти (CLAUDE.md, RAG, MCP-сервер памяти) и не понимаете, почему он не помогает или даже мешает. Статья даёт критерий, что вообще стоит складывать в память, а что модель и так знает.

**Как.** Практический фильтр от автора: храните только то, что отсутствует в весах модели и что нельзя безопасно вывести из общих знаний — факты, привязанные к конкретному пользователю, кодовой базе, клиенту, моменту. Пример из статьи: модель и так знает, что миграции часто ломаются, но не может знать, что «биллинг-воркер этой команды молча зависит от устаревшего Redis-ключа» — вот это и есть кандидат в память. Триггер на извлечение тоже переформулирован: не «семантически похоже?», а «ответит ли модель уверенно неправильно без этого факта?». Три слоя предлагается не смешивать: grounding (актуальные доки/API/конфиги), операционные знания (устройство и конвенции системы), эпизодическая память (что произошло в этой сессии).

**Подвох.** Это эссе с выводами, а не готовый инструмент: бенчмарк автора (250 задач, OrKa Brain против baseline) дал всего +0,12 по рубрике и упёрся в 74% позиционного смещения оценщика — то есть прирост на грани шума. Чёткий сигнал виден только на «длинных последовательностях в одном домене». Берите как принцип проектирования, не как доказанный результат.

`mcp` · Источник: [blog/marcosomma](https://dev.to/marcosomma/the-model-does-not-need-memory-the-situation-does-196g)

## 5. Как собрать MCP-сервер на Flama

**Зачем.** Если вы пишете свой MCP-сервер на Python, чтобы дать Claude Code доступ к своим функциям/данным, и не хотите вручную возиться с JSON-RPC. Flama даёт декораторы для tools/resources/prompts поверх ASGI-API.

**Как.** Установка: `uv pip install "flama[pydantic]"`. Минимальный сервер из статьи: создать `app = Flama(...)`, зарегистрировать эндпойнт `app.mcp.add_server("/mcp/tools/", "tools", version="2.0.0")`, объявить инструмент декоратором `@app.mcp.tool("add", description="Add two integers", mcp="tools")` над обычной Python-функцией. Запуск: `flama run app:app`. Из коробки есть фоновые задачи (`task=True`), интерактивный ввод (Elicitation) и UI-шаблоны; сервер — stateless JSON-RPC поверх HTTP, масштабируется горизонтально.

**Подвох.** Статья НЕ показывает, как подключить готовый сервер именно к Claude Code — она доводит только до тестирования через `curl`, пример регистрации в `claude mcp add` или `claude_desktop_config` придётся искать самому. Транспорт здесь — HTTP/JSON-RPC, тогда как многие клиенты ждут stdio; учтите это при интеграции. Туториал свежий и нишевый (11 реакций).

`mcp` · Источник: [blog/vortico](https://dev.to/vortico/building-an-mcp-server-with-flama-2ad9)

## 7. samvallad33/vestige — когнитивная память для AI-агентов на Rust

**Зачем.** Если Claude Code при отладке упорно ищет «похожую» ошибку, а не настоящую причину поломки. Vestige заточен под каузальный поиск: дотягивается до тихого изменения, которое сломало сегодняшнюю сборку, даже если оно не пересекается по словам с симптомом — там, где обычный векторный поиск бесполезен.

**Как.** Установка: `npm install -g vestige-mcp-server@latest`; проверка — `vestige-mcp --version` и `vestige stats`. Подключение к Claude Code: `claude mcp add vestige vestige-mcp -s user` (для остальных редакторов — блок `{"mcpServers":{"vestige":{"command":"vestige-mcp"}}}`). Даёт агенту 13 инструментов (recall, backfill, smart_ingest, graph, suppress — активное забывание и др.) и 3D-граф памяти на `http://localhost:3927/dashboard`. Бинарник 23 МБ, полностью офлайн, один раз качает embedding-модель ~130 МБ.

**Подвох.** Заявленные цифры («60% recall против 0% у чистого вектора») — от автора, без независимой проверки; ★565 и явного раздела ограничений в README нет. На Intel-маках сборка требует динамической линковки с ONNX Runtime из Homebrew (Microsoft перестал собирать x86_64-macOS после 1.23.0) — на старом железе установка усложняется.

`mcp` · Источник: [github/samvallad33](https://github.com/samvallad33/vestige)

## 8. Бесплатный браузерный инструмент оценки безопасности MCP-конфигурации

**Зачем.** Если у вас в Claude Code/Cursor накопились MCP-серверы и вы хотите быстро проверить конфиг на типовые дыры, прежде чем давать агенту реальные права. Полезно перед тем, как подключать чужой MCP-сервер или выкатывать настройку на команду.

**Как.** Открыть `https://alih552.github.io/mcp-forge/scan.html` и вставить свой конфиг (поддерживаются форматы Claude Desktop, Cursor и обычный `.mcp.json`). Инструмент локально, в браузере, выдаёт оценку 0-100 и группирует находки по 8 категориям: отсутствие аутентификации у remote-серверов, открытый HTTP вместо HTTPS, секреты в открытом виде, незакреплённые версии в `npx`/`uvx` (тянут latest), слишком широкий доступ к ФС (`/` или домашняя папка), отключённая проверка TLS (`NODE_TLS_REJECT_UNAUTHORIZED=0`), биндинг на `0.0.0.0`, и оценка расхода токенов на сервер. К каждому пункту — конкретный фикс.

**Подвох.** Это статический разбор конфига по эвристикам, а не аудит самого кода MCP-серверов — он не поймает вредоносную логику внутри инструмента, только небезопасную настройку. Проект совсем свежий и малозамеченный (2 реакции); проверяйте выводы руками, не доверяйте оценке вслепую.

## 14. archcore-ai/plugin — плагин для context engineering в Claude Code и Cursor

**Зачем.** Если Claude Code пишет код «не туда» и игнорирует конвенции команды, потому что каждый раз не знает архитектуру репо. Archcore даёт агенту типизированный, запрашиваемый слой знаний (решения в Git, правила, паттерны, спеки), чтобы новые правки ложились в нужные места без повторения инструкций.

**Как.** Установка CLI: `curl -fsSL https://archcore.ai/install.sh | bash` (Windows — `irm https://archcore.ai/install.ps1 | iex`), проверка `archcore --version`; CLI работает как MCP-сервер и должен быть в PATH до установки плагина. Рабочий цикл по слэш-командам: `/archcore:init` (засеять правила стека и импортировать существующие CLAUDE.md/AGENTS.md/.cursorrules) → `/archcore:context` (подгрузить релевантные правила и спеки перед правкой файла) → `/archcore:capture` (задокументировать недокументированный модуль/API) → `/archcore:plan` → `/archcore:decide` (записать решение и превратить в правило) → `/archcore:audit` (найти устаревшую/расходящуюся документацию).

**Подвох.** Авторы прямо очерчивают границы: это НЕ память чата, НЕ библиотека промптов и НЕ one-shot генерация спеки в код — только «слой правды о репо». Установка идёт через `curl | bash` (скачивание и исполнение скрипта с чужого домена — оцените риск). Проект совсем маленький (★48), зрелость низкая.

## 15. Show HN: TypeScript7 LSP — плагин Claude Code

**Зачем.** Если вы работаете в Claude Code над крупным TypeScript-проектом и встроенный TS-language-server тормозит на диагностике и подсказке типов. Плагин подменяет его нативным Go-сервером TypeScript 7 — быстрее типизация и диагностика, что напрямую ускоряет работу агента с кодом.

**Как.** Последовательность из README дословно: `npm install -g typescript@rc` → `/plugin marketplace add mjn298/ts7-lsp-plugin` → `/plugin install ts7-lsp@ts7-lsp-marketplace` → `/plugin disable typescript-lsp@claude-plugins-official` → `/reload-plugins`. Под капотом — нативный сервер TypeScript 7 (`typescript@rc`), запускаемый как `tsc --lsp --stdio`, вместо Node-обёртки `typescript-language-server` над классическим `tsserver`.

**Подвох.** Завязка на release-candidate TypeScript (`typescript@rc`) — поведение может меняться, для прод-проектов нестабильно.
