## 1. Claude Code — дайджест недели 26: claude mcp login, ответ на !-команды, /rewind до /clear

**Зачем.** Тем, кто скриптует MCP и гоняет фоновые сабагенты: собраны точечные изменения за неделю с готовыми командами, а не общий обзор.

**Как.** `claude mcp login <name>` / `claude mcp logout <name>` прогоняют OAuth сервера без интерактивного меню `/mcp`; префикс `!` теперь даёт ответ на вывод (`! npm test` → разбор падений, отключается `respondToBashCommands:false`); `/rewind` восстанавливает разговор из состояния до `/clear`; новый `sandbox.credentials` запрещает песочнице читать секреты; фоновые сабагенты показывают запрос прав в основной сессии вместо авто-отказа.

**Подвох.** Ответ на `!`-команду стоит как обычный промпт (лишние токены). Часть настроек (`autoMode.classifyAllShell`, OTEL-логи ответов) — для продвинутых и по умолчанию выключены.

`mcp · subagents · automation · tips` · Источник: [docs/Anthropic](https://code.claude.com/docs/en/whats-new/2026-w26)

## 2. Claude Code — дайджест недели 25: права по параметрам инструмента и /config из промпта

**Зачем.** Командам, которым нужна тонкая настройка прав и безопасный авто-режим в CI: конкретный синтаксис управления сабагентами и настройками.

**Как.** Deny/ask-правила вида `Tool(param:value)` — например `Agent(model:opus)`, чтобы запретить дорогие Opus-сабагенты; `/config key=value` (в т.ч. в `-p`-режиме) меняет настройку без редактирования файла; авто-режим по умолчанию блокирует деструктивные `git` и `terraform/pulumi destroy`; Артефакты публикуют живую страницу сессии.

**Подвох.** В параметрическом матчинге прав легко ошибиться синтаксисом; авто-блокировку деструктивных команд можно обойти явным запросом — не полагаться как на единственную защиту.

`subagents · automation · workflows · tips` · Источник: [docs/Anthropic](https://code.claude.com/docs/en/whats-new/2026-w25)

## 3. Что менять при переходе на Sonnet 5 по умолчанию в Claude Code (гайд билдера)

**Зачем.** Тем, кто ведёт общие конфиги команды и следит за расходом: объясняет, что поменять при переходе на Sonnet 5 и новый MCP-флоу подтверждений.

**Как.** Новые сессии — на Sonnet 5 с адаптивным thinking; ручной extended thinking и sampling-параметры убраны, токенайзер новый (пересчитать `max_tokens`). MCP: серверы из закоммиченного `.mcp.json` в недоверенном воркспейсе теперь требуют явного approve. Организации задают дефолтные модели на уровне орг/роли.

**Подвох.** Цена за токен не изменилась, но «тот же промпт стоит дороже» из-за нового токенайзера — предупредить команду заранее, иначе счёт вырастет незаметно.

## 5. 8 продакшн-процессов Claude Code с правилом, когда они окупаются

**Зачем.** Командам, уставшим от one-shot промптов: готовые паттерны плюс правило, когда они реально окупаются.

**Как.** Ключевое правило: задача <30 мин — просто промпт; 30 мин–часы — план-затем-билд (`/plan {id}` → апрув → `/task-start` → `/review` → `/task-done`). Остальное: read-only аудит в `AUDIT.md`, общий `REFACTOR.md` для мульти-репо, файловые локи + `HANDOFFS.md` для параллельных агентов, `UPGRADE.md` для зависимостей.

**Подвох.** Паттерны требуют дисциплины скоупинга и ревью-гейтов; на мульти-репо и рое агентов координационные накладные расходы могут перевесить пользу. Шаблоны надо адаптировать под свой домен.

`workflows · automation · refactoring · testing` · Источник: [blog/Taha](https://www.ayautomate.com/blog/best-claude-code-workflows)

## 6. Как мониторить расход токенов Claude Code и где они утекают

**Зачем.** Инженерным командам, у которых счёт за Claude Code растёт непонятно куда, — как измерить и где утекает.

**Как.** Подписка — `/usage` и `/status`; API — `/cost`. Историю разбирает `ccusage` (фильтры по дате/проекту/модели), реалтайм — `Claude-Code-Usage-Monitor` (burn rate, время до лимита). Топ-драйверы расхода: раздутый вывод инструментов (один grep ≈ 10–15k токенов), размытые промпты с переделками, сабагенты (+200–500% оверхеда), Agent Teams, extended thinking (по умолчанию, ×5 к input).

**Подвох.** Локальные логи сессий хранятся лишь 30 дней по умолчанию — для долгой аналитики заранее поднять `cleanupPeriodDays` в настройках.

`cost-tokens · tips` · Источник: [blog/Faros AI](https://www.faros.ai/blog/claude-code-token-usage)

## 7. Как Anthropic сдерживает собственные кодинг-агенты — и почему это трудно повторить

**Зачем.** Security-командам, которым нужно масштабировать контроль над кодинг-агентами на весь парк инструментов, а не только Claude Code.

**Как.** Три слоя: (1) Environment — процесс/VM-песочницы, egress-allowlist, режимы монтирования ФС, изоляция кредов (Anthropic выложил `sandbox-runtime` с профилями Seatbelt/bubblewrap); (2) Model — системные промпты, классификаторы, флоу апрувов, защита от prompt-injection; (3) External — аудит MCP-серверов, инспекция вывода инструментов, отношение к клонированным репо как к враждебному вводу.

**Подвох.** Anthropic может так защищаться, потому что контролирует весь стек; у большинства организаций зоопарк агентов (Cursor, Copilot, Gemini CLI) с несогласованными политиками — централизованное управление парком пока не решено.

`automation · tips` · Источник: [blog/Highflame](https://www.highflame.com/blog/how-anthropic-contains-its-own-coding-agents-and-get-that-coverage-across-your-fleet/)

## 8. Crew: параллельные сессии Claude Code видят друг друга и переписываются

**Зачем.** Тем, кто гоняет несколько Claude Code одновременно по одному коду и хочет, чтобы агенты не мешали друг другу, а обменивались статусом.

**Как.** `npm install -g @0xmmo/crew` либо плагином `/plugin marketplace add 0xmmo/crew`. Каждая сессия получает контекст-блок с активными сессиями; `crew send <target> "..."` шлёт сообщение; команда `crew` показывает статусы, выжимки и куски транскриптов. Ноль конфигурации, авто-встраивается в настройки Claude Code.

**Подвох.** Node ≥18, только macOS/Linux; спящую сессию нельзя разбудить извне (сообщения ждут до 24 ч); отображение ограничено 5 записями на сессию и 8 сессиями; читает сессии, но не меняет их.

`subagents · workflows` · Источник: [github/0xmmo](https://github.com/0xmmo/crew)

## 9. harness: Rust control plane для флота агентов с governance и кросс-ревью

**Зачем.** Тем, кто оркестрирует «флот» агентов и хочет governance: единый жизненный цикл задач, независимое ревью (агент не ревьюит сам себя), политики исполнения.

**Как.** `cargo build --release -p harness-cli`, `bash scripts/doctor.sh`, `./start-server.sh`, проверка `curl http://127.0.0.1:9800/health`. Нужны Rust 1.88+, Postgres 14+, GitHub-токен и хотя бы один рантайм агента (Claude CLI / Codex CLI / Anthropic API). Есть режим MCP-сервера (JSON-RPC) и экспорт OpenTelemetry.

**Подвох.** HTTP API «fails closed»: без `HARNESS_API_TOKEN` (или явного `allow_unauthenticated`) не поднимется. Инфраструктурно тяжёлый (Postgres, сборка на Rust), зрелость средняя — последний релиз конца мая.

`workflows · automation` · Источник: [github/majiayu000](https://github.com/majiayu000/harness)

## 10. Brick-SR1: роутер, отправляющий запрос Claude Code на самую дешёвую подходящую модель

**Зачем.** Тем, кто жжёт бюджет, гоняя всё на дорогой модели: роутер сам выбирает модель под сложность запроса.

**Как.** `git clone`, `cd brick-SR1/apps/cli && npm install && npm run build && npm link`, `brick claude on`, затем в НОВОЙ сессии выбрать `brick-claude` в `/model`. Роутер оценивает промпт по 6 осям (код, синтез, следование инструкциям, матан, планирование, знания) и направляет в пул моделей с учётом цены. Режимы eco/mid/max привязаны к слайдеру усилий.

**Подвох.** Бета (интеграция с Codex сырая), Docker-дистрибутив только с v2.1.0 — пока собирать из исходников; добавляет свой прокси-слой между вами и моделью.
