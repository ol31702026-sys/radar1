## 1. Claude Code 2.1.202: настройка размера динамических воркфлоу + OTel-атрибуты запусков

**Зачем.** Тем, кто гоняет многоагентные динамические воркфлоу и хочет управлять их масштабом, стоимостью и наблюдаемостью.

**Как.** Обнови до v2.1.202; в /config выбери «Dynamic workflow size» (small/medium/large — контролирует число агентов). В OTel-телеметрию агентов, порождённых воркфлоу, добавлены атрибуты workflow.run_id и workflow.name — по ним восстанавливается активность одного запуска. Также фиксы: inline Ctrl+R history search, /rename фоновых сессий, mTLS-handshake.

**Подвох.** Точечный релиз (в основном багфиксы); настройка влияет только на динамические воркфлоу, не на обычные сабагенты.

`workflows · automation · tips` · Источник: [github/Anthropic](https://github.com/anthropics/claude-code/releases/tag/v2.1.202)

## 4. Claude Code 2.1.200: AskUserQuestion больше не автопродолжается, режим прав по умолчанию — Manual

**Зачем.** Тем, кого раздражало, что агент «проскакивал» вопросы и действовал без явного согласия; безопаснее для прод-репозиториев.

**Как.** Обнови до v2.1.200; по умолчанию AskUserQuestion ждёт ответа (автопродолжение включается в /config); дефолтный permission mode = «Manual»; команда /review <pr> снова однопроходная (быстрее).

**Подвох.** Больше ручных подтверждений = медленнее автономные прогоны; кто полагался на авто-продолжение, должен явно включить его обратно.

## 7. Stash: общая память для кодинг-агентов — ускорил длинные сессии Claude Code на 49%

## 9. Live Memory: постоянная память репозитория, чтобы агент перестал перечитывать файлы

**Зачем.** Тем, кто теряет токены на повторном чтении одних и тех же файлов в длинных сессиях.

**Как.** Подними HTTP-MCP-сервер (`cd server && python -m live_memory`), затем `/plugin marketplace add https://github.com/shofer-dev/claude-code-live-memory`, `/plugin install live-memory@shofer-live-memory`. Хуки пассивно «подслушивают» читаемые файлы; спрашивай `ask_live_memory`.

**Подвох.** Нужен Python ≥3.10 и запущенный сервер (HTTP-транспорт не авто-стартует); память привязана к workspace (cwd) и не переносится; на чисто-редакторских задачах экономия ~нулевая.

`mcp · cost-tokens` · Источник: [github/shofer-dev](https://github.com/shofer-dev/claude-code-live-memory)

## 11. Официальный MCP-сервер Anthropic в песочнице gVisor: 60+ атак, разбор изоляции

**Зачем.** Тем, кто пускает сторонние/непроверенные MCP-серверы и думает об изоляции сверх обычного Docker.

**Как.** `docker run --runtime=runsc --network none --read-only --cap-drop ALL ...` — gVisor перехватывает syscalls до реального ядра. Проверено 6 категорий атак (path traversal, SSRF, инъекции, prompt injection): ни сетевых вызовов, ни записи в чувствительные пути, ни доступа к секретам.

**Подвох.** Автор честно отмечает: для этого «воспитанного» сервера gVisor не поймал ничего, чего не поймал бы обычный Docker. Ценность — защита от будущих вредоносных серверов, эксплуатирующих ядро.

`mcp · testing` · Источник: [blog/edison_flores](https://dev.to/edison_flores_6d2cd381b13/i-ran-anthropics-official-mcp-server-in-a-gvisor-sandbox-heres-what-happened-a6j)

## 13. VNX: governance-оркестрация CLI-агентов (Claude/Codex/Gemini) с квитанциями и quality-gates

**Зачем.** Командам, которым нужен аудируемый след и governance над параллельными агентами, а не «чёрный ящик» автономного мерджа.

**Как.** `pip install vnx-orchestration`, затем `vnx init`, `vnx migrate`, `vnx doctor`, `vnx dispatch-agent --agent name`. Dual-LLM adversarial review (Codex+Gemini), человек подтверждает мерджи; память слоями PAST/CURRENT/FUTURE.

**Подвох.** Worktree-изоляция выключена по умолчанию и не гарантирует race-free при параллели; часть фич — Tier 2/3 (не достроено); «это не sandbox» — только аудит-след; нужен установленный CLI на PATH.
