## 1. Claude Code и Cowork приходят в госсектор США (публичная бета, FedRAMP High)

**Зачем.** Если вы в организации с требованиями по compliance (госсектор, регулируемые отрасли), это показывает, какой набор контролей делает агентного помощника пригодным для чувствительных сред: не «выключить AI», а обвязать его аудитом и лимитами.

**Как.** Доступ через Claude for Government Desktop (та же кодовая база, что у коммерческого продукта, но в FedRAMP High). Ключевое: hash-chained (защищённый от подмены) аудит-лог всех административных действий; жёсткие лимиты трат и выбора модели по тиру мест; история переписки хранится локально на управляемом устройстве, инференс — внутри FedRAMP-периметра. Запрос доступа — на claude.com/solutions/government.

**Подвох.** Публичная бета; документы по безопасности (FedRAMP Secure Configuration Guide, отчёты пентестов) доступны только через trust center под NDA. Для рядового разработчика вне госсектора — скорее ориентир по governance, чем инструмент.

## 2. Китай выпустил предупреждение о «бэкдоре» в Claude Code (версии 2.1.91–2.1.196)

**Зачем.** Полезно тем, кто оценивает риски использования агентных инструментов в компании: разобраться, что именно вменяют Claude Code и как реагировать (обновление, контроль сетевого трафика dev-инструментов).

**Как.** Национальная база уязвимостей КНР (CNVDB) указала на «встроенный механизм мониторинга», который якобы собирал location/identity-данные и слал их без явного согласия. Затронуты версии 2.1.91 (2 апреля) — 2.1.196 (29 июня). Рекомендация: немедленно обновиться до последней безопасной версии либо удалить; усилить контроль доступа и мониторить исходящий трафик dev-инструментов.

**Подвох.** Это заявление госоргана в напряжённом геополитическом контексте, а не независимый техотчёт; Anthropic на момент публикации «не ответила оперативно» на запрос комментария. Проверяйте на первоисточниках.

## 3. Как выбирать модель vs уровень усилий (effort) в Claude Code

**Зачем.** Практично для ежедневной работы и контроля счёта: понять, когда переключить на модель побольше, а когда просто поднять effort — вместо того чтобы вслепую тратить токены на топовую модель на любой мелочи.

**Как.** Ключевая эвристика: спросите себя — Claude «не знал достаточно» (проблема модели) или «не постарался» (проблема effort)? Модель побольше — для реально сложного (тонкие баги, незнакомый домен, архитектурные решения). Выше effort — когда контекст ясен, но Claude пропустил файлы, не гонял тесты, бросил многошаговую задачу. По умолчанию: начинать с дефолтов, effort — общее предпочтение, а не микроменеджмент.

**Подвох.** Это концептуальный гайд, а не таблица «модель X = задача Y»: конкретный выбор всё равно требует чутья. Более высокая модель/effort = дороже и медленнее.

`cost-tokens · prompting · tips` · Источник: [blog/Anthropic](https://claude.com/blog/claude-model-and-effort-level-in-claude-code)

## 4. State of AI Coding 2026: Codex делает меньше багов, но большинство сидит на Claude

**Зачем.** Даёт цифры для трезвого взгляда: не «какой инструмент лучше», а какой компромисс между качеством и скоростью выбирают команды. Полезно, если решаете, стоит ли менять инструмент или процесс.

**Как.** По данным cubic (сотни тысяч коммитов с трекингом AI-авторства): Claude Code — основной агент у ~80% разработчиков (Cursor 18% и падает, Codex 17%); Opus — 80% основного использования по неделям. При этом GPT-5.5 даёт меньше багов на изменённые строки. Продуктивность: медиана PR 6→12 в месяц, размер PR 56→124 строк. Вывод: выигрыш в объёме перевешивает небольшой проигрыш по багам.

**Подвох.** Выборка смещённая — только команды на ревью-платформе cubic с включённым трекингом; это ранние паттерны, не весь рынок.

## 6. fable5-orchestrator: токен-экономная многоагентная оркестрация с guard-хуками

**Зачем.** Для тех, кто хочет оставить топ-модель «за главного» без слива лимита: маршрутизация по тирам моделей + механическая дисциплина, чтобы агент не разбегался и не бросал требования недоделанными.

**Как.** `/plugin marketplace add Rylaa/fable5-orchestrator`, затем `/plugin install orchestrator@fable-orchestrator`. Fable 5/Opus планирует, Sonnet 5 реализует. `Requirements Ledger` (`.workflow/LEDGER.md`) — чекбоксы требований; guard-хуки бьют по spawn (нет делегирования без леджера) и close (нельзя закрыть сессию с незакрытыми пунктами).

**Подвох.** Хуки проверяют существование леджера, а не качество; пункт можно пометить готовым без проверки. На одном экспериментальном backend async-spawn обходил guard. Неофициальный проект.

`subagents · hooks · cost-tokens` · Источник: [github/Rylaa](https://github.com/Rylaa/fable5-orchestrator)

## 7. pdf-reader-mcp: разбор PDF для агента с доказательной привязкой к странице

**Зачем.** Когда Claude Code работает с PDF (контракты, спеки, отчёты) и важно, чтобы он не галлюцинировал, а ссылался на конкретное место документа.

**Как.** `claude mcp add pdf-reader -- npx @sylphx/pdf-reader-mcp` (Node.js ≥22.13, работает локально). Три инструмента: `read_pdf`, `search_pdf`, `pdf_evidence`. OCR для сканов с word-level provenance; trust-report флагит скрытый текст, prompt-injection, спуфинг и небезопасные ссылки.

**Подвох.** Нужны абсолютные пути, загрузки по URL по умолчанию нет. Качество по сканам зависит от OCR-конфигурации; тяжёлый визуальный анализ — provider-dependent.

## 8. agent-memory-hub: общая память Claude Code между сессиями на своём Supabase

**Зачем.** Больной вопрос — Claude «забывает» между сессиями. Это self-hosted способ дать агенту сквозную память, причём данные в вашем собственном Supabase.

**Как.** Клон репо → проект Supabase → миграции (`sql/01-schema.sql`) → креды в `.env` → хуки на SessionStart/Stop/SessionEnd в `settings.json` → `./scripts/setup.sh`. Другие машины: клон + тот же `.env` + setup. Есть слой «фактов» с confidence decay и опциональный семантический поиск.

**Подвох.** Извлечение фактов и профилирование требуют своей модели (Ollama/Gemini/OpenAI). Правила профиля не применяются автоматически — нужен ручной ревью. Семантический поиск требует pgvector и Edge Function.

## 10. safer-dependencies: скиллпак, проверяющий пакеты до того, как Claude их добавит

**Зачем.** Агент любит поставить первый попавшийся пакет — а это вектор supply-chain атак. Скиллпак ловит типосквоты и заброшенные/уязвимые пакеты до `npm install`/`pip install`.

**Как.** `python3 .../safer_dependencies_manager.py interactive_install` — установщик пропишет хуки в `.claude/settings.json`. Пять режимов (Normal/Intercept/Pre-Install/Post-Install/Post-Agent) и пять проверок: провенанс, возраст версии (≥7 дней), CVE через OSV, хеш-пиннинг PyPI, health-check. Покрывает npm, PyPI, RubyGems, Maven, Go, Rust, PHP.

**Подвох.** Лицензия closed-source (бесплатно для личного/внутреннего). Транзитивные CVE ловятся, но не чинятся автоматически. Правки манифеста через `sed -i` обходят Intercept. Fail-open: при ошибке операция проходит.

`skills` · Источник: [github/robert-auger](https://github.com/robert-auger/safer-dependencies)

## 12. Инструменты vs сырые команды: теория токеновой стоимости MCP

**Зачем.** Полезно, если вы навешали MCP-серверов и удивляетесь счёту. Статья объясняет, где MCP оправдан, а где сырая команда дешевле.

**Как.** Суть: схемы MCP занимают 2000+ токенов на сервер и подмешиваются в каждый запрос, даже если инструмент не используется; CLI выигрывает по выводу при фильтрации (460 против 3110 токенов). Гибрид: запросы — через MCP-инструменты, операции с побочными эффектами — через сырой CLI. Один практик срезал траты на ~60% ($8.64→$3.45/день).

**Подвох.** Бенчмарки сторонние, абсолютные числа зависят от модели и цен. «Gateway pattern» требует своей инфраструктуры. Оптимизация исходит из того, что токены важнее латентности.

`mcp · cost-tokens` · Источник: [blog/ev3lynx727](https://dev.to/ev3lynx727/tools-vs-raw-commands-the-token-cost-theory-d1g)

## 13. Prompt → Context → Loop Engineering: что реально поменялось

**Зачем.** Помогает перестать «крутить формулировки» и посмотреть на агентную работу как на цикл с проверяющим (verifier) — что важнее для тех, кто строит workflow/loop в Claude Code.

**Как.** Три сдвига: prompt (2022–24, единица «сообщение») → context (~2025, единица «контекстное окно») → loop (июнь 2026, единица «цикл проверки»). Ключ — «проблема верификатора»: детерминированная проверка (тесты, запрещённые импорты) даёт стабильный вердикт; модель-грейдер («понятно ли объяснение?») наследует ненадёжность. Перед постройкой цикла явно решите: проверка про корректность или про качество?

**Подвох.** «Разрыв истории решений»: циклы не хранят рассуждения между сессиями, и отклонённый подход уверенно переоткрывается, пока обе проверки «зелёные». Статья концептуальная — готовых конфигов не даёт.

`prompting` · Источник: [blog/cleverhoods](https://dev.to/reporails/prompt-engineering-context-engineering-loop-engineering-what-actually-changed-2357)

## 14. Кейс: правительство Альберты просканировало 466 млн строк кода Claude Code за 20 часов

**Зачем.** Редкий крупный госкейс с цифрами: как связка «rules engine + Claude Code» масштабируется на безопасность огромного парка систем и что это даёт по времени/деньгам. Ориентир для тех, кто думает про массовый аудит легаси.

**Как.** ~50 агентов работали автономно и параллельно. Двухстадийно: rules engine флагит известные паттерны, затем Claude Code анализирует флаги и цитирует конкретные файлы и строки для верификации разработчиком. Масштаб: 466 млн строк, 1280 приложений, 3400 репозиториев за ~20 часов (ручной эквивалент ~6.5 лет). 25-летний Java-портал переписан за 4–5 дней. Опубликованы 21 open-source техпейпер (Velocity White Papers).

**Подвох.** Официальный кейс — нет разбора провалов и ложных срабатываний, только успехи. Экономические оценки («>6 лет», «$2 млрд») — прикидки самой команды, не независимый аудит.

## 15. Кейс: Claude Code в соревнованиях Kaggle по поиску на графах Кэли

**Зачем.** Живой пример разделения труда «человек — стратегия, агент — реализация» в исследовательской задаче, а не в CRUD-приложении. Полезно понять, где Claude Code реально ускоряет, а где упирается.

**Как.** Задача — поиск по графам Кэли в двух соревнованиях (IHES Picture Cube, Megaminx). Claude написал по сути все тренировочные скрипты, варианты beam-search, порты на TPU, автоматизацию сабмитов и постобработку; автор ревьюил код и принимал экспериментальные решения. Результаты: IHES 42 718 → 23 224 хода; Megaminx 79 522 (у конкурента 93 606).

**Подвох.** Автор прямо говорит: агент «плохо держит неоднозначность и предпочитает безопасные идеи», уходил в бесконечные итерации, а сложную распределённую TPU-логику не осилил — пришлось звать Codex после 10 попыток. Прорывы шли от человеческой постановки.

`case-study` · Источник: [blog/Andrey Lukyanenko](https://andlukyane.com/blog/cayleypy-kaggle-with-claude)
