## 2. Китай выпустил предупреждение о «бэкдоре» в Claude Code (версии 2.1.91–2.1.196)

**Зачем.** Полезно тем, кто оценивает риски использования агентных инструментов в компании: разобраться, что именно вменяют Claude Code и как реагировать (обновление, контроль сетевого трафика dev-инструментов).

**Как.** Национальная база уязвимостей КНР (CNVDB) указала на «встроенный механизм мониторинга», который якобы собирал location/identity-данные и слал их без явного согласия. Затронуты версии 2.1.91 (2 апреля) — 2.1.196 (29 июня). Рекомендация: немедленно обновиться до последней безопасной версии либо удалить; усилить контроль доступа и мониторить исходящий трафик dev-инструментов. Контекст: Anthropic ранее (в 2.1.198, 1 июля) убрала отдельную стеганографическую систему для детекции distillation-атак.

**Подвох.** Это заявление госоргана в напряжённом геополитическом контексте, а не независимый техотчёт; Anthropic на момент публикации «не ответила оперативно» на запрос комментария. Проверяйте на первоисточниках, не принимайте формулировки за установленный факт.

## 3. Как выбирать модель vs уровень усилий (effort) в Claude Code

**Зачем.** Практично для ежедневной работы и контроля счёта: понять, когда переключить на модель побольше, а когда просто поднять effort — вместо того чтобы вслепую тратить токены на топовую модель на любой мелочи.

**Как.** Ключевая эвристика: спросите себя — Claude «не знал достаточно» (проблема модели) или «не постарался» (проблема effort)? Модель побольше — для реально сложного (тонкие баги, незнакомый домен, архитектурные решения, когда модель «уверенно неправа» при хорошем контексте). Выше effort — когда контекст ясен, но Claude пропустил файлы, не гонял тесты, бросил многошаговую задачу на полпути. По умолчанию: начинать с дефолтов и относиться к effort как к общему предпочтению, а не микроменеджить его на каждую задачу.

**Подвох.** Это концептуальный гайд, а не таблица «модель X = задача Y»: конкретный выбор всё равно требует чутья и зависит от вашей кодовой базы. Более высокая модель/effort = дороже и медленнее.

## 4. State of AI Coding 2026: Codex делает меньше багов, но большинство сидит на Claude

**Зачем.** Даёт цифры для трезвого взгляда: не «какой инструмент лучше», а какой компромисс между качеством и скоростью выбирают команды. Полезно, если решаете, стоит ли менять инструмент или процесс.

**Как.** По данным cubic (сотни тысяч коммитов с трекингом AI-авторства): Claude Code — основной агент у ~80% разработчиков (Cursor 18% и падает, Codex 17% стабильно); Opus-модели — 80% основного использования по неделям. При этом GPT-5.5 даёт меньше багов на изменённые строки. Продуктивность выросла: медиана PR 6→12 в месяц, размер PR 56→124 строк; у команд с трекингом AI пишет 90–100% кода (было 78% в апреле). Вывод: выигрыш в объёме перевешивает небольшой проигрыш по багам.

**Подвох.** Выборка смещённая — только команды на ревью-платформе cubic с включённым трекингом AI-атрибуции; это ранние паттерны, не весь рынок. Абсолютные числа зависят от методологии.

## 6. fable5-orchestrator: токен-экономная многоагентная оркестрация с guard-хуками

**Зачем.** Для тех, кто хочет оставить топ-модель «за главного» без слива лимита: маршрутизация по тирам моделей + механическая дисциплина, чтобы агент не разбегался и не бросал требования недоделанными.

**Как.** `/plugin marketplace add Rylaa/fable5-orchestrator`, затем `/plugin install orchestrator@fable-orchestrator` (или вручную скопировать Python-хуки в `~/.claude/hooks/` и слить настройки в `~/.claude/settings.json`). Fable 5/Opus планирует и принимает финальные решения, Sonnet 5 делает реализацию/тесты/объём. `Requirements Ledger` (`.workflow/LEDGER.md`) — чекбоксы требований до делегирования; guard-хуки бьют по двум точкам: spawn (нет большого делегирования без леджера) и close (нельзя закрыть сессию с незакрытыми пунктами). MIT, есть pytest-покрытие.

**Подвох.** Хуки проверяют существование леджера, а не его качество — поверхностный леджер пройдёт; пункт можно пометить выполненным без реальной проверки. Детект модели — по строке (форсится через env `FABLE_ORCH_PROFILE`). На одном экспериментальном backend async-spawn обходил guard. Неофициальный проект без даты релиза.

## 7. pdf-reader-mcp: разбор PDF для агента с доказательной привязкой к странице

**Подвох.** Нужны абсолютные пути к файлам, загрузки по URL по умолчанию нет. Качество по сканам зависит от вашей OCR-конфигурации; тяжёлый визуальный анализ — provider-dependent, не встроен. Маркетинговые «39/39 release-gate checks» — внутренние; реальная надёжность зависит от сложности ваших PDF.

## 8. agent-memory-hub: общая память Claude Code между сессиями на своём Supabase

**Зачем.** Больной вопрос — Claude «забывает» между сессиями. Это self-hosted способ дать агенту сквозную память (решения, предпочтения, процедуры), причём данные у вас в собственном Supabase, а не в чужом облаке.

**Как.** Клонировать репо → создать бесплатный проект Supabase → прогнать миграции (`sql/01-schema.sql`) → прописать креды в `.env` → добавить хуки на три события в `settings.json` (SessionStart, Stop, SessionEnd) → `./scripts/setup.sh`. Другие машины: клон + тот же `.env` + setup, и сессии синхронизируются. Есть слой «фактов» с confidence decay и опциональный семантический поиск (гибрид keyword + смысл).

**Подвох.** Для ядра LLM не нужен, но извлечение фактов и профилирование требуют своей модели (Ollama/Gemini/OpenAI). Правила профиля и дедуп фактов не применяются автоматически — нужен ручной ревью. Семантический поиск (Phase 2) требует pgvector и Edge Function. Без даты релиза на странице.

## 10. safer-dependencies: скиллпак, проверяющий пакеты до того, как Claude их добавит

**Зачем.** Агент любит поставить первый попавшийся пакет — а это вектор supply-chain атак. Скиллпак ловит типосквоты и заброшенные/уязвимые пакеты до `npm install`/`pip install`, автоматически через хуки.

**Как.** Быстрый старт: клонировать репо и запустить `python3 .../safer_dependencies_manager.py interactive_install` — установщик пропишет хуки в `.claude/settings.json` и добавит команды в allowlist. Пять режимов: Normal (ручной аудит), Intercept (правит уязвимые версии после записи манифеста), Pre-Install (блок рискованных пинов до установки), Post-Install (транзитивные CVE в lock-файле), Post-Agent (аудит манифестов от субагентов). Пять проверок: провенанс, возраст версии (берёт стабильную, опубликованную ≥7 дней назад), CVE через OSV, хеш-пиннинг PyPI, health-check. Актуальная версия v0.5.2.

**Подвох.** Лицензия closed-source: бесплатно для личного/внутреннего использования, но монетизация самого инструмента требует платной лицензии. Pre-Install видит только прямые пины; транзитивные ловит Post-Install, но не чинит автоматически. Правки манифеста через `sed -i`/скрипты обходят Intercept. Fail-open по дизайну: при ошибке (нет Python, сбой сети) операция проходит — защита деградирует, а не блокирует.

`skills` · Источник: [github/robert-auger](https://github.com/robert-auger/safer-dependencies)

## 14. Кейс: фрилансер зарабатывает ~$4200/мес на автоматизации рутины малому бизнесу через Claude Code

**Зачем.** Практичная модель для того, кто хочет зарабатывать на автоматизации для малого бизнеса, а не просто «уметь Claude Code». Показывает, что продаётся: сокращение повторяющейся ручной работы сотрудников, а не технология сама по себе.

**Как.** Разделение труда между моделями: Claude Code — основной «строитель» (код, подключение API, генерация документации, рефакторинг логики автоматизаций — за счёт лучшего удержания контекста); Gemini — дешёвая классификация, извлечение и массовая обработка текста. Реальные заказы: (1) для перепродажной точки — автопереименование фото, переписывание описаний товаров, форматирование листингов и синхронизация остатков между площадками; (2) очередь обработки сообщений из Facebook/почты/форм с классификацией на Gemini, валидацией формата и confidence-скорингом, который решает, нужен ли человек, — чтобы обрабатывать больше заявок без найма. Процесс: сначала нарисовать поток информации и найти разрывы, потом строить с логированием, мониторингом, валидацией и retry. ## 15. Кейс: соло-основатель собрал прод-SaaS OnboardingHub только на Claude Code (вечерами за 8 недель)

**Зачем.** Показывает потолок для соло-билдера: не MVP-однодневка, а продакшн-SaaS с биллингом, ролями и документацией, собранный «в свободные вечера» параллельно с фултаймом. Ориентир по реальному соотношению «часы человека / результат» и по тому, где всё ломается на проде.

**Как.** Автор написал «Agent Operating Manual» (CLAUDE.md), задающий процесс: сначала архитектура и дизайн документируются, потом фичи идут через Linear-тикеты; параллельные сессии агента в изолированных окружениях; ревью-гейты в стиле DHH с явным подтверждением перед мержем. Claude Opus 4.5 (позже 4.6 на полировке) — соавтор почти каждого коммита. Масштаб: 713 коммитов, 38 632 строки в 657 файлах за ~8 недель (15 дек 2025 – 8 фев 2026). Готовый продукт: мультитенант с ролевым доступом, 2FA и CAPTCHA, биллинг Stripe с подписками, медиатека с транскодингом видео, почта с шаблонами, дашборд аналитики, сайт документации на 29 страниц, деплой на Heroku. Оценка: ~25–45 часов человека против ~800 часов соло-разработки — «20–30x рычаг».

**Подвох.** Самое ценное — честный разбор аварии на проде 3–4 февраля: перепутанные db:migrate/db:prepare, не созданные таблицы Solid Queue (async-джобы молча падали), забытые env-переменные Stripe, обновление aws-sdk-s3 сломало загрузки в Cloudflare R2 сменой алгоритма контрольных сумм, нехватка памяти на дино 512 МБ. Всё чинилось 14 «форвардными» коммитами без откатов. Вывод: агент отлично пишет код, но инфраструктуру, миграции и секреты придётся держать под личным контролем.
