## 1. Claude Code 2.1.208–209: режим для скринридеров, vim-ремапы и чинилка фоновых агентов

**Зачем.** Тем, кто держит Claude Code обновлённым: часть фиксов чинит болезненные баги — контекст «сбрасывался» после автообновления CLI, фоновые агенты теряли текст при сбое доставки, правка файла ломалась, если файл меняли после чтения.

**Как.** Проверить версию — `claude --version`. Экранный диктор: `claude --ax-screen-reader` / `CLAUDE_AX_SCREEN_READER=1` / `"axScreenReader": true`. В vim-режиме двухклавишные ремапы вставки задаёт `vimInsertModeRemaps` (`jj`→Escape). Корпоративный лаунчер — `CLAUDE_CODE_PROCESS_WRAPPER`. Большие таблицы (200+ строк) больше не вешают вывод.

**Подвох.** Это ченджлог, а не гайд: конкретику применения ищите в доках. Auto mode по умолчанию — только на Bedrock/Vertex/Foundry.

`tips · ide-integration` · Источник: [docs/Anthropic](https://code.claude.com/docs/en/changelog#2-1-208)

## 3. Пошаговый subagent-воркфлоу для Claude Code: research → tests → review → verify

**Зачем.** Тем, кто заметил, что один агент «тупеет» по мере заполнения контекста: детальную работу выносят специализированным субагентам, а главный тред держат чистым. Для реальных фич с тестами и ревью.

**Как.** Воркфлоу в `~/.claude/CLAUDE.md` + `~/.claude/skills/dev-workflow/SKILL.md`, роли в `.claude/agents/`: `deep-explore` (чтение кода → `research-<task>.md`), `test-author`, `implementer`, `code-reviewer` (не правит код), `code-review-judge`, `verifier`. Каждый агент пишет результат в файл; ревью-петля крутится до нуля critical/major.

**Подвох.** Автор сам «ещё учится», гайд быстро устареет. Пайплайн жжёт токены и избыточен для тривиальных правок (для них — фаст-трек).

`subagents · workflows` · Источник: [blog/Rajiv Prabhakar](https://software.rajivprab.com/2026/07/13/using-subagents-to-improve-claude-code/)

## 4. Fleet Deck: борд для нескольких параллельных сессий Claude Code на одной машине

**Зачем.** Тем, кто гоняет несколько агентов разом и путается, в какой вкладке что происходит: борд централизует статусы, разрешения и вопросы, ловит конфликты и оживляет упавшие сессии.

**Как.** `claude plugin marketplace add lacion/fleet-deck` → `claude plugin install fleetdeck@fleetdeck`; либо `npm install -g fleetdeck` → `fleetdeck service install` → `fleetdeck service start`. Node 22.5+, борд на 127.0.0.1:4711. Предупреждает о правке одного файла в пределах 30 мин; батч-спавн агентов в отдельных worktree; «revive» для упавших.

**Подвох.** v0.8.0, ~22 звезды — молодой проект (тестирован на CLI 2.1.206–207). Спавн = реальный биллинг API. Реле разрешений не работает с headless (`-p`); для спавна/терминала нужен tmux.

`workflows · automation · tips` · Источник: [github/lacion](https://github.com/lacion/fleet-deck)

## 6. Themis: self-hosted ревью PR на Claude/Codex/GLM с обучением на ваших правках

**Зачем.** Небольшим командам, кому нужен автоматический ревьюер на своей инфраструктуре и со своими ключами/моделью, а не облачный SaaS: читает «доктрину ревью» из `.themis/`.

**Подвох.** Очередь задач в памяти (теряется при рестарте); нужен Claude **Max** (не Pro); для Codex иногда `THEMIS_CODEX_SANDBOX=danger-full-access`. v0.4.0, MIT.

## 7. MCP как «запутанный помощник»: инъекция инструкций и DNS-rebinding против агентов

**Зачем.** Всем, кто подключает MCP к Claude/Claude Code: как через «легитимный» вывод инструмента или подсунутую веб-страницу в агента попадают чужие инструкции и как атакующий добирается до внутренней сети.

**Как (защита).** Белые списки URL для fetch; изоляция MCP-серверов с блокировкой RFC1918 (10.x/192.168.x) против DNS-rebinding; обязательное подтверждение человеком для разрушающих вызовов; детект-правила на подозрительные результаты инструментов и DNS-аномалии.

**Подвох.** Это модель угроз, а не патч: у MCP нет встроенной аттестации происхождения — меры организационные/сетевые, внедрять руками. Официальный fetch-сервер не санитизирует контент.

`mcp` · Источник: [blog/Diego Diaz](https://dev.to/alejandxr/the-mcp-confused-deputy-provenance-gaps-instruction-injection-and-dns-rebinding-in-the-model-918)

## 8. Local MCP: даёт Claude доступ к Mail, iMessage, Teams и файлам на Mac — без облака

**Зачем.** Фрилансерам и малым командам, кто хочет автоматизировать рутину поверх личных данных (почта, мессенджеры, календарь, документы), не выгружая их в чужое облако.

**Как.** Поставить приложение на macOS 13+ (Apple Silicon/Intel), выбрать интеграции — ключи/OAuth для личных данных не требуются, бесплатный старт. Опциональный «Cloud Relay» (шифрованный WebSocket) даёт удалённый доступ, например с телефона.

**Подвох.** Только macOS (другие ОС «в роадмапе»). WhatsApp — неофициальный клиент (риск блокировки); Cloud Relay требует доверия к реле; независимого аудита безопасности нет.

## 9. Наблюдаемость для ИИ-эры: MCP + маскирование PII и само-починка от алерта до PR

**Зачем.** Командам, кто хочет пустить ИИ-агента в наблюдаемость, но боится утечки персональных данных: как искать по логам без плейнтекста PII и как замкнуть инцидент в автоматический фикс.

**Как.** PII хешируется HMAC-SHA256 и на записи, и на поиске — запрос по не-PII идентификатору резолвится в e-mail уже внутри MCP-сервера, модель плейнтекст не видит; вывод MCP маскируется (`r***@air-closet.com`). Цепочка: alert → релей → агент тянет логи по MCP → PR-фикс → авто-мёрдж по зелёному CI.

**Подвох.** Ротация HMAC-ключа не реализована; что считать ошибкой — на гайдлайнах и ревью без автопринуждения; поисковый эндпоинт уязвим к probe-атакам авторизованных пользователей.

`mcp · automation · case-study` · Источник: [blog/Ryosuke Tsuji](https://dev.to/ryantsuji/observability-design-for-the-ai-era-reconciling-pii-protection-with-ai-searchability-and-driving-2f0l)

## 10. Sx 2.0: делиться скиллами Claude через папку Dropbox — без git и терминала

**Зачем.** Нетехническим командам (маркетинг, юристы, продажи, операции), кто создаёт полезные скиллы, но не хочет возиться с командной строкой и версионированием.

**Как.** Открыть приложение, указать библиотеку на общую облачную папку (Dropbox/Drive/OneDrive/iCloud), перетащить скиллы как markdown. Коллеги указывают ту же папку и жмут Sync — приложение переводит ассеты в формат каждого ИИ-клиента и ставит локально.

**Подвох.** Автор сам сомневается в «реальной латентности Dropbox» — надёжность распределённой синхронизации на масштабе не проверена. Это продукт-обёртка, а не официальный механизм Claude.

## 11. Как количественно оценить качество промпта для Claude Code — без вызова модели

**Зачем.** Тем, кто хочет объективно сравнивать и улучшать промпты и скиллы: базовые признаки — Objective (глаголы действия), Grounding (пути к файлам, имена функций), Direction (критерии и ограничения) — плюс шесть обогащающих.

**Как.** Геометрическое среднее базовых признаков + бонусы за обогащение и штрафы против «накрутки» (капы на ключевые слова, штраф за повтор биграмм до 40%, минимум слов). Открытый инструмент h5i (460+ звёзд) ставится и запускается локально, выдаёт 0–100 и флаги.

**Подвох.** Правило-основанная эвристика: игнорирует сложность задач, зависит от ручных списков ключевых слов, не ловит логические противоречия и уязвима к оптимизации под метрику.

`prompting` · Источник: [blog/Hideaki Takahashi](https://medium.com/@Koukyosyumei/how-to-quantitatively-evaluate-prompt-quality-in-claude-code-and-codex-47f2f27f4bc5)

## 12. Исследование: у адоптеров Claude Code/Copilot CLI в Microsoft на ~24% больше смёрдженных PR

**Зачем.** Тем, кто внедряет ИИ-агентов в команде и хочет данных, а не хайпа: что предсказывает удержание и как распространять инструмент — через видимость у коллег, а не приказом сверху.

**Как (выводы).** Наблюдение 4 месяца: (1) первое использование расходится по социальным связям внутри компании; (2) удержание лучше предсказывает уровень активности кодинга, а не демография; (3) адоптеры мёрджат ~на 24% больше PR, прирост стабилен.

**Подвох.** Авторы оговаривают: смёрдженный PR — прокси выпуска, «не равно ценности». Это крупный энтерпрайз (Microsoft), не малый бизнес — цифры не переносите на соло-команду.

`case-study · tips` · Источник: [arxiv/Murphy-Hill et al.](https://arxiv.org/abs/2607.01418)

## 13. aftr: делать видео в After Effects кодом — Claude Code как режиссёр (OSS)

**Зачем.** Соло-креаторам и мелким студиям, кто хочет генерировать и рендерить моушн-видео из текста, не кликая вручную в After Effects.

**Как.** Без AE: `git clone https://github.com/Arman-Luthra/aftr.git && npm install && npm run build:jsx && npm test`. С AE: `npm run deploy:panel` + `npm run controller`, затем Window → Extensions → aftr. ~90 команд (композиции, слои, эффекты, рендер), one-call VFX-пресеты, склейка через FFmpeg. Claude Code берёт команды aftr как инструменты.

**Подвох.** Нужен лицензионный After Effects 2024–2026, Node 18+, FFmpeg. Модальные диалоги ExtendScript вешают мост; предпросмотра в реальном времени нет. Репозиторий ранний (~9 звёзд).

## 14. Свой Claude Code-агент 24/7 на VPS с управлением из Telegram — за 60–90 минут

**Зачем.** Соло-предпринимателю или небольшой команде, кому нужен «работник» на связи из телефона: ставить задачи и получать результаты из Telegram, пока тяжёлое крутится на своём сервере, а не на ноутбуке.

**Как.** VPS Debian 12 ($12–36/мес), нативный Claude Code + Bun + Node 22, авторизация подпиской (Pro/Max). Бот через BotFather, плагин `claude --channels plugin:telegram@claude-plugins-official`, пара по 6-значному коду, доступ по allowlist. Персистентность: `tmux` + systemd/cron; скиллы в `~/.claude/skills/`; `/loop` для повторов.

**Подвох.** Задачи `/loop` живут 7 дней и работают, лишь пока Claude Code запущен; после ребута состояние теряется. Сообщения во время простоя **дропаются**. `--dangerously-skip-permissions` — только для изолированного личного использования. Итог ~$32–56/мес.

`automation · case-study` · Источник: [blog/Parash Panta](https://www.dplooy.com/blog/build-a-247-claude-code-agent-with-telegram-control)

## 15. Как соло-разработчик собрал сайт с CMS на Claude Code — без фреймворков (PHP, ваниль)

**Зачем.** Соло и малому бизнесу, кто хочет понять реальный расклад «человек решает архитектуру — Claude пишет код»: где выигрыш по скорости, а где всё равно нужен ваш контроль.

**Как.** Жёсткие правила заранее: никаких backend/frontend/CSS-фреймворков. Против «забывания» решений между сессиями — `CLAUDE.md` + 15 файлов-правил. Спринты по 30–45 минут (длиннее — «дрейф контекста»). Итог: ~48,7к строк PHP, 40 миграций, CMS с админкой; «задачи на день делались за час-два».

**Подвох.** Claude тянет к переусложнению и дублирует CSS без строгих правил; на длинных сессиях «иногда просто переставал работать» — спасал рестарт. Вывод: Claude Code не заменяет понимание того, что строишь, а убирает рутину. (Пост от февраля, обновлён 14 июля.)

`case-study` · Источник: [blog/Ivan Mišić](https://ivanmisic.net/blog/ai-tools/how-i-built-this-site-with-claude-code)
