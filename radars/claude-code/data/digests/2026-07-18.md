## 2. repo-harness: файловый «каркас» для устойчивых сессий Claude Code и Codex

**Зачем.** Когда одну задачу тянешь несколько сессий и агент каждый раз заново «изучает» проект и теряет договорённости — harness держит план, задачи и критерии приёмки в файлах репо.

**Как.** Установка: `curl -fsSL https://raw.githubusercontent.com/Ancienttwo/repo-harness/main/install.sh | sh`, затем `repo-harness install --profile minimal` и `repo-harness adopt`. Хуки перехватывают события сессии (start/edit/prompt/stop) и обновляют файлы `.ai/`, `plans/`, `tasks/`, `docs/spec.md`; есть карточки ревью pass/fail и прогрессивная загрузка контекста для экономии токенов.

**Подвох.** По умолчанию подсказки не блокирующие (advisory). Тяжёлые проверки (Unity, E2E, мобильные симуляторы) сам не гоняет — нужны ручные манифесты верификации. Версия ранняя (v0.10.1).

`workflows · tips` · Источник: [github/Ancienttwo](https://github.com/Ancienttwo/repo-harness)

## 3. RegressionLedger: хук, который не даёт Claude Code повторять уже провалившиеся правки

**Зачем.** Знакомая боль: агент в цикле применяет один и тот же неудачный патч. RegressionLedger разрывает петлю.

**Как.** Ставится как плагин: `/plugin marketplace add anlor1002-alt/regressionledger` → `/plugin install regressionledger@anlor1002-plugins` (или `npx regressionledger init`). Механика: снимает «отпечаток» изменённого кода, после прогона тестов помечает правку pass/fail, а на PreToolUse блокирует повтор похожей проваленной правки с сообщением вида «ты это уже пробовал 2 часа назад, оно упало с [ошибка] — смени подход». Режимы `block`/`warn`, порог схожести настраивается.

**Подвох.** При падении сразу нескольких правок атрибуция приблизительная; флаки-тесты могут обвинить невиновный фикс (лечится `minFailures: 2`). Пока только Claude Code.

`hooks · testing` · Источник: [hn/anlor1002-alt](https://github.com/anlor1002-alt/regressionledger)

## 4. weave-claude-code: трейсинг сессий, вызовов инструментов и сабагентов в W&B Weave

**Зачем.** Кому нужна наблюдаемость за агентом: почему сессия столько стоила, что делали сабагенты, какой инструмент завис — всё видно структурированно.

**Как.** `npm install -g weave-claude-code`, затем `weave-claude-code install` — установщик пропишет плагин и спросит проект Weave и W&B API-ключ. Дальше каждая сессия логируется без правок кода.

**Подвох.** Плагин отправляет наружу данные сессии: промпты, ответы, входы/выходы инструментов, пути и содержимое файлов. Очистки PII и редактирования секретов пока НЕТ — на чувствительных проектах ставить не стоит.

## 5. retok: анализатор расхода токенов для Claude Code и Codex CLI (без зависимостей)

**Зачем.** Чтобы понять, почему счёт за токены такой, и увидеть конкретные точки перерасхода (повторное кэширование после истечения TTL, раздутый контекст, циклы ретраев).

**Как.** `git clone https://github.com/d-date/retok && cd retok && ./retok` (последние 30 дней), либо `./retok --days 7`, `--json`, `--lang ja`. Читает `~/.claude/projects` и `~/.codex/sessions`, выдаёт оценку затрат, cache hit rate и рекомендации вроде «повторное кэширование после TTL: ~135M токенов ($873)». Нужен только Python 3.7+.

**Подвох.** Стоимость — оценка по встроенной таблице цен (может отставать от актуальных). Нестандартные пути логов задаются вручную.

`cost-tokens` · Источник: [hn/opwizardx](https://github.com/d-date/retok)

## 7. claudeisms: список слов, которые Claude Code переупотребляет

**Зачем.** Кто редактирует за Claude тексты, доки и коммиты и хочет убирать узнаваемые ИИ-штампы — вот готовый список «маркеров».

**Как.** Автор сравнил ~140 МБ своих сессий (Opus 4.7/4.8) с 1.72 млрд слов из Stack Overflow и выделил статистически переупотребляемые слова: лидер — «load-bearing» (в 7500 раз чаще). Используйте список как стоп-слова при вычитке или просите Claude заменять их на точные формулировки.

**Подвох.** Данные — один пользователь и один стиль работы; сам анализ делал Claude (не перепроверено). Актуально для Opus 4.7/4.8, у других моделей иначе.

## 8. tmux-agent-tabs: статус агентских сессий Claude Code прямо во вкладках tmux

**Зачем.** Кто гоняет несколько сессий Claude Code параллельно в tmux и теряет, какая ждёт ответа, а какая упала.

**Как.** Через TPM: `set -g @plugin 'jhickner/tmux-agent-tabs'`, затем `prefix + I`. Добавляешь плейсхолдеры `#{agent_tab_icon}` в `window-status-format` и `#{agent_usage}` в `status-right`. Индикаторы: спиннер (работает), `?` (ждёт ввода), `e` (ошибка), `●` (фоновая задача готова); хоткей `M-a` прыгает на вкладку, требующую внимания; опциональный звук по завершении.

**Подвох.** Для точного статуса нужна настройка хуков в Claude Code; для полосы квоты — `curl`, для Grok — `jq`. Кэш статуса переписывается каждую секунду — при облачной синхронизации конфига задайте `@agent_tabs_dir`.

`ide-integration · tips` · Источник: [hn/jhickner](https://github.com/jhickner/tmux-agent-tabs)

## 11. «Anatomy of a Misfeature»: почему авто-ответ на AskUserQuestion в Claude Code — плохая идея

**Зачем.** Кто гоняет автономные или параллельные сессии и не хочет, чтобы агент принимал решения за вас, пока вы отошли.

**Как.** В 2.1.198 инструмент `AskUserQuestion` при бездействии 60 сек продолжает сам («No response after 60s — continued without an answer»). Автор советует: держать фичу opt-in, требовать её описания в release notes, а до фикса — отключить авто-обновление через `"DISABLE_AUTOUPDATER": "1"` в settings.json.

**Подвох.** При нескольких параллельных агентах за таймерами не уследить, а частичные ответы досабмичиваются вместе с догадками модели. Это мнение автора — но проблема реальная и прилетает молча через авто-апдейт.

`tips` · Источник: [hn/oalders](https://www.olafalders.com/2026/07/17/claude-code-anatomy-of-a-misfeature/)

## 12. ontology-atlas: локальная память-онтология для Claude Code через MCP

**Зачем.** Кому надоело, что агент каждую сессию заново «въезжает» в проект — граф хранит только значимые доказательства реализации (файлы, тесты, команды, подтверждающие владение доменом), и всё это в git-diff'ах на ревью.

**Как.** `npx ontology-atlas init ./ontology`, затем `ontology-atlas analyze . --vault ./ontology`, `bootstrap`, `workspace-brief`/`agent-brief`. Одно хранилище — три интерфейса: macOS-приложение (Tauri), CLI и MCP-сервер (24 JSON-RPC инструмента) для интеграции с агентом. Агент читает граф перед кодом, смотрит только релевантные доказательства, потом предлагает обновления памяти markdown-диффами.

**Подвох.** Это НЕ серверная граф-БД и не полный индекс символов — структурный поиск делегируется language server / grep / Serena, атлас держит только «значимое». Локально, без облака и логина.

`mcp · workflows` · Источник: [github/wlsdks](https://github.com/wlsdks/ontology-atlas)

## 14. Кейс: ревью-воркфлоу на Claude Code, экономящий ~30 минут на инженера в день

**Зачем.** Небольшой продуктовой команде, которая тонет в ревью и хочет снять рутинный первый проход, не жертвуя качеством.

**Как.** Claude Code запускается автоматически при открытии PR как первый ревьюер. Вместо одного агента — фан-аут на узкие роли: корректность (краевые случаи, nil, конкурентность), консистентность (нейминг, идиомы обработки ошибок), тесты (покрытие vs реальные изменения), доступность/i18n. Каждая находка обязана ссылаться на файл/строку и описывать конкретный сценарий сбоя — чтобы не было шума.

**Подвох.** Честные грабли автора: авто-мёрдж не взлетел (агенты оптимизировали «тесты зелёные», а не корректность — кнопку мёрджа оставили людям); один общий агент находил меньше багов, чем несколько узких; базовые метрики надо было снять ДО внедрения, а не восстанавливать потом.

`workflows · testing · case-study` · Источник: [blog/Oleh Veheria](https://veheria.tech/blog/claude-code-code-review-workflow/)
